# 브랜치 생성, 전환 및 삭제

---

## 👨‍💻 실전 프로젝트: 브랜치 생성부터 삭제까지

브랜치의 생성, 전환, 삭제를 하나의 완결된 시나리오로 실습해보겠습니다. 아래 실습은 실제 개발 현장에서 브랜치를 다루는 전형적인 패턴을 보여줍니다. 이 실습을 통해 브랜치의 전체 생애주기를 경험하고 브랜치 이름 짓는 규칙까지 자연스럽게 익힐 수 있습니다.

```bash
# 1. 실습 저장소 생성
$ mkdir branch-lifecycle && cd branch-lifecycle
$ git init
$ echo "# Branch Lifecycle" > README.md
$ git add . && git commit -m "초기 설정"

# 2. 기능 개발 브랜치 생성 및 전환 (한 번에)
$ git switch -c feature/user-login

# 3. 기능 개발 작업 수행
$ echo "<form>아이디: <input type='text'/></form>" > login.html
$ git add . && git commit -m "로그인 폼 HTML 추가"
$ echo "body { background: #fff; }" > style.css
$ git add . && git commit -m "로그인 페이지 스타일 추가"

# 4. main 브랜치로 전환
$ git switch main

# 5. 브랜치 목록 확인
$ git branch -v
* feature/user-login a1b2c3d 로그인 페이지 스타일 추가
  main              d4e5f6g 초기 설정

# 6. main 브랜치에도 변경사항 발생
$ echo "v1.0" > version.txt
$ git add . && git commit -m "버전 파일 추가"

# 7. feature/user-login 브랜치 병합 (3장에서 자세히 다룸)
$ git merge feature/user-login

# 8. 병합 완료된 브랜치 삭제
$ git branch -d feature/user-login

# 9. 최종 브랜치 목록 확인
$ git branch
* main
```

위 실습에서 주목할 점은 브랜치 생성(`git switch -c`), 브랜치 전환(`git switch main`), 브랜치 삭제(`git branch -d`)라는 브랜치 관리의 3대 핵심 명령어를 모두 사용했다는 것입니다. 브랜치 이름을 `feature/user-login`과 같이 `feature/` 접두사를 사용하여 목적을 명확히 한 것도 좋은 네이밍 관행입니다. 이처럼 일관된 브랜치 네이밍 규칙은 팀 협업에서 코드의 의도를 파악하는 데 큰 도움이 됩니다.

---

## 학습 목표

- `git branch` 명령어를 사용하여 브랜치 목록을 확인하고 새 브랜치를 생성할 수 있습니다.
- `git switch`와 `git checkout`을 사용하여 브랜치를 전환하는 방법을 이해합니다.
- 더 이상 필요하지 않은 브랜치를 `git branch -d` 및 `-D` 옵션으로 삭제할 수 있습니다.
- 원격 브랜치를 푸시하고 삭제하는 방법과 브랜치 전환 시 주의할 점을 학습합니다.

브랜치를 자유자재로 다루는 것은 Git 사용의 핵심입니다. 우리가 여러 작업을 동시에 진행하면서도 코드를 안정적으로 유지할 수 있는 것은 브랜치 생성, 전환, 삭제를 자유롭게 할 수 있기 때문입니다. 이 문서에서는 브랜치를 생성하고, 다른 브랜치로 전환하며, 불필요한 브랜치를 삭제하는 방법을 배웁니다. 이를 통해 우리는 효율적이고 체계적인 브랜치 관리 능력을 갖추게 될 것입니다. 이 장에서 다루는 명령어들은 Git을 사용하는 모든 개발자가 매일같이 사용하는 기본 명령어이므로, 반드시 숙지하시기 바랍니다.

---

## 1. 브랜치 목록 확인: `git branch`

현재 저장소에 있는 모든 브랜치의 목록을 확인합니다. `git branch` 명령어는 단순히 브랜치 목록을 보여주는 것 이상으로 다양한 옵션을 제공합니다. 특히 `-v`, `-vv`, `-r`, `-a`와 같은 옵션을 조합하면 저장소의 전체 브랜치 현황을 상세히 파악할 수 있습니다.

```bash
git branch
```

**출력 예시:**
```
* main
  feature/login
  feature/payment
```

`*` 표시가 된 브랜치가 현재 작업 중인 브랜치입니다. 이 표시는 HEAD 포인터가 해당 브랜치를 가리키고 있음을 의미합니다. 따라서 현재 어떤 브랜치에서 작업 중인지 항상 확인하는 것이 좋은 습관입니다.

**브랜치 목록 자세히 보기:**
```bash
$ git branch -v              # 브랜치 + 마지막 커밋 메시지
* main        a1b2c3d [최신] README 업데이트
  feature/x   d4e5f6f 로그인 기능 구현
  feature/y   g7h8i9j 결제 모듈 추가

$ git branch -vv             # 추적 브랜치 정보까지
* main        a1b2c3d [origin/main] README 업데이트
  feature/x   d4e5f6f [origin/feature/x: ahead 2] 로그인 기능 구현

$ git branch -r              # 원격 브랜치만
  origin/main
  origin/develop
  origin/feature/x

$ git branch -a              # 로컬 + 원격 모든 브랜치
* main
  feature/x
  remotes/origin/main
  remotes/origin/develop
```

`-v` 옵션은 각 브랜치의 마지막 커밋 메시지를 함께 보여주므로 브랜치의 작업 내용을 빠르게 파악할 수 있습니다. `-vv` 옵션은 로컬 브랜치가 어떤 원격 브랜치를 추적하고 있는지, 그리고 로컬이 원격보다 몇 커밋 앞서 있는지(ahead) 또는 뒤쳐져 있는지(behind)까지 표시해줍니다. 이 정보는 협업 과정에서 매우 유용하게 활용됩니다.

## 2. 새 브랜치 생성: `git branch <이름>`

```bash
git branch feature/readme-update
```

이 명령어는 현재 커밋 위치를 기준으로 새 브랜치를 생성합니다. 브랜치를 생성만 할 뿐, 현재 작업 브랜치가 자동으로 전환되지는 않습니다. 이 점을 반드시 기억해야 합니다. 브랜치를 생성한 후에도 HEAD는 여전히 기존 브랜치를 가리키고 있으므로, 새 브랜치에서 작업하려면 별도로 전환 명령어를 실행해야 합니다. 만약 생성과 동시에 전환까지 한 번에 수행하고 싶다면, 이후에 설명할 `git switch -c` 명령어를 사용하면 됩니다.

---

## 3. 브랜치 전환: `git switch` (또는 `git checkout`)

다른 브랜치로 이동하여 작업하려면 브랜치를 전환합니다. 브랜치를 전환하면 작업 디렉토리의 파일들이 해당 브랜치의 마지막 커밋 상태로 변경됩니다. 이는 Git이 내부적으로 HEAD 포인터를 이동시키고, 해당 커밋의 스냅샷으로 작업 디렉토리를 갱신하기 때문입니다.

**`git switch` 사용 (Git 2.23+ 버전에서 권장):**
```bash
git switch feature/readme-update
```

**`git checkout` 사용 (전통적인 방법):**
```bash
git checkout feature/readme-update
```

Git 2.23 버전부터 `git switch`와 `git restore` 명령어가 도입되었습니다. 기존의 `git checkout` 명령어는 브랜치 전환과 파일 복원이라는 두 가지 역할을 동시에 수행하여 초보자에게 혼란을 주곤 했습니다. `git switch`는 브랜치 전환 전용 명령어로, 그 목적이 명확하여 실수를 줄일 수 있습니다. 새로운 프로젝트를 시작한다면 `git switch` 사용을 권장합니다.

지금까지 브랜치를 생성하고 전환하는 기본적인 방법을 알아보았습니다. 이제 브랜치 생성과 전환을 한 번에 수행하는 방법을 학습해보겠습니다.

---

## 4. 브랜치 생성과 동시에 전환

앞서 배운 `git branch` 생성과 `git switch` 전환을 각각 수행하는 대신, 이 두 단계를 하나의 명령어로 결합할 수 있습니다. 이 방법은 특히 새 브랜치를 만들자마자 바로 작업을 시작해야 할 때 매우 편리합니다. 실제 개발 현장에서는 대부분의 경우 이 방식으로 브랜치를 생성하고 전환합니다.

**`git switch -c` 사용 (권장):**
```bash
git switch -c feature/new-function
```

**`git checkout -b` 사용:**
```bash
git checkout -b feature/new-function
```

**브랜치 이름 짓는 규칙 예시:**
```bash
# 기능 개발
git switch -c feature/login-page         # 새로운 로그인 페이지
git switch -c feature/user-profile       # 사용자 프로필 기능

# 버그 수정
git switch -c hotfix/crash-on-login      # 로그인 시 크래시 버그
git switch -c hotfix/typo-in-readme      # README 오타 수정

# 실험/리팩토링
git switch -c refactor/database-layer    # DB 계층 리팩토링
git switch -c experiment/new-algorithm   # 새 알고리즘 실험
```

브랜치 이름을 지을 때는 `타입/설명` 형식을 따르는 것이 일반적입니다. `feature/`는 새로운 기능 개발을, `hotfix/`는 긴급 버그 수정을, `refactor/`는 코드 리팩토링을 의미합니다. 이처럼 일관된 네이밍 규칙은 저장소를 탐색할 때 각 브랜치의 목적을 즉시 파악할 수 있게 해줍니다. 또한 슬래시(`/`)를 사용하면 Git에서 브랜치를 계층적으로 구조화하여 관리할 수 있어 편리합니다.

지금까지 브랜치 생성과 전환 방법을 학습하였습니다. 이제 더 이상 필요하지 않은 브랜치를 삭제하는 방법에 대해 알아보겠습니다.

---

## 5. 브랜치 삭제: `git branch -d <이름>`

더 이상 필요하지 않은 브랜치는 삭제할 수 있습니다. 브랜치 삭제는 완료된 기능의 브랜치를 정리하여 저장소를 깔끔하게 유지하는 중요한 관리 작업입니다.

```bash
git branch -d feature/readme-update
```

**참고:** 현재 작업 중인 브랜치는 삭제할 수 없습니다. 다른 브랜치로 전환한 후 삭제해야 합니다. 이는 Git의 안전장치로, 현재 작업 중인 브랜치를 실수로 삭제하는 것을 방지합니다. 만약 현재 브랜치를 삭제하려고 시도하면 "Cannot delete branch '...' checked out at '...'"와 같은 오류 메시지를 보게 됩니다.

**강제 삭제 (`-D`):** 병합되지 않은 브랜치라도 강제로 삭제하려면 대문자 `-D` 옵션을 사용합니다. 소문자 `-d`는 병합이 완료된 브랜치만 삭제하지만, 대문자 `-D`는 병합 여부와 관계없이 강제로 삭제합니다. 단, 이 작업은 되돌릴 수 없으므로 신중하게 사용해야 합니다. 브랜치를 강제 삭제하면 해당 브랜치의 커밋들도 Git의 참조에서 사라지게 됩니다.

```bash
git branch -D feature/abandoned-work
```

## 6. 원격 브랜치 관련 명령어

로컬 브랜치 작업을 원격 저장소에 반영하려면 `git push`를 사용합니다. 로컬에서 작업한 브랜치를 원격 저장소에 공유해야 팀원들이 함께 사용할 수 있습니다. 또한 원격 브랜치가 더 이상 필요 없을 때는 삭제하여 원격 저장소를 정리할 수 있습니다.

```bash
# 로컬 브랜치를 원격에 푸시 (원격에 같은 이름의 브랜치 생성)
git push origin feature/login

# 로컬 브랜치를 삭제하고 원격 브랜치도 삭제하려면
git push origin --delete feature/login
```

원격 브랜치를 푸시할 때 `git push origin feature/login` 명령어를 사용하면 로컬의 `feature/login` 브랜치가 원격 저장소의 `feature/login` 브랜치로 푸시됩니다. 이때 `-u` 옵션(`git push -u origin feature/login`)을 함께 사용하면 추적 브랜치가 설정되어, 이후에는 `git push`만으로도 해당 브랜치를 푸시할 수 있습니다. 원격 브랜치를 삭제할 때는 `git push origin --delete` 뒤에 삭제할 브랜치 이름을 지정하면 됩니다.

지금까지 브랜치 관련 기본 명령어를 학습하였습니다. 이제 실제 실습 예제를 통해 브랜치를 만들고 전환하는 과정을 직접 체험해보겠습니다.

## 실습 예제: 브랜치 만들고 전환하기

```bash
# 현재 브랜치 확인
git branch
* main

# 새 브랜치 생성
git branch experiment

# 브랜치 목록 확인 (experiment가 생성됨)
git branch
  experiment
* main

# experiment 브랜치로 전환
git switch experiment

# experiment 브랜치에서 파일 수정 후 커밋
echo "실험적인 코드" >> test.txt
git add test.txt
git commit -m "experiment 브랜치에서 작업"

# 다시 main 브랜치로 전환 (test.txt 파일이 사라진 것처럼 보임)
git switch main
```

이렇게 브랜치 전환 시 작업 디렉토리의 파일 내용이 해당 브랜치의 마지막 커밋 상태로 변경됩니다. 이것이 브랜치의 핵심 동작 방식입니다. experiment 브랜치에서 만든 `test.txt` 파일은 main 브랜치로 전환하면 보이지 않게 되는데, 이는 main 브랜치의 마지막 커밋에는 해당 파일이 존재하지 않기 때문입니다. 반대로 experiment 브랜치로 다시 전환하면 `test.txt` 파일이 다시 나타납니다. 이러한 동작은 각 브랜치가 완전히 독립된 작업 공간임을 보여줍니다.

## 브랜치 전환 시 주의할 점

브랜치를 전환할 때는 항상 작업 디렉토리의 상태를 확인해야 합니다. 커밋되지 않은 변경 사항이 있는 상태에서 브랜치를 전환하려고 하면 Git이 전환을 거부할 수 있습니다. 이는 커밋되지 않은 작업이 손실되는 것을 방지하기 위한 Git의 안전장치입니다.

```bash
# ⚠️ 커밋하지 않은 변경 사항이 있으면 브랜치 전환이 거부될 수 있음
$ git switch feature/login
error: Your local changes to the following files would be overwritten by checkout:
    index.html
Please commit your changes or stash them before you switch branches.

# 해결 방법 1: 변경 사항 커밋
$ git add . && git commit -m "임시 저장"

# 해결 방법 2: stash (변경 사항을 임시 보관)
$ git stash
Saved working directory and index state WIP on main: a1b2c3d...

$ git switch feature/login  # 이제 가능!

# stash 복원
$ git stash pop
```

두 가지 해결 방법 중 어떤 것을 선택할지는 상황에 따라 다릅니다. 변경 사항이 완료되었다면 커밋하는 것이 가장 깔끔합니다. 하지만 작업이 아직 완료되지 않았거나 커밋하기 애매한 상황이라면 `git stash`를 사용하여 변경 사항을 임시 보관하는 것이 좋습니다. `stash`는 변경 사항을 스택에 저장해두고 나중에 `git stash pop`으로 다시 꺼내올 수 있는 편리한 기능입니다.

지금까지 브랜치 전환 시 주의할 점을 학습하였습니다. 이제 실제 개발 상황과 유사한 시나리오를 통해 브랜치 생성, 전환, 병합, 삭제의 전 과정을 실습해보겠습니다.

## 실습 예제: 로그인 기능 개발 시나리오

실제 개발 현장에서 발생할 수 있는 복합적인 시나리오를 통해 브랜치 관리의 전 과정을 실습해보겠습니다. 이 시나리오는 기능 개발 중 긴급 버그가 발생하여 핫픽스 브랜치를 만들고, 버그 수정 후 다시 원래 기능 개발로 복귀하는 전형적인 개발 패턴을 담고 있습니다.

```bash
# 1. main 브랜치에서 시작
git switch main

# 2. 로그인 기능 개발 브랜치 생성
git switch -c feature/login
# (여기서 로그인 기능 구현...)
echo "<form>Login</form>" > login.html
git add . && git commit -m "로그인 폼 추가"

# 3. 긴급 버그 발견! main으로 돌아가기
git switch main
# (버그 수정 브랜치 생성)
git switch -c hotfix/critical-bug
echo "버그 수정" > fix.txt
git add . && git commit -m "긴급 버그 수정"

# 4. 버그 수정 완료, main에 병합
git switch main
git merge hotfix/critical-bug
git branch -d hotfix/critical-bug

# 5. 다시 로그인 기능 개발로 복귀
git switch feature/login
echo "<button>Submit</button>" >> login.html
git add . && git commit -m "로그인 버튼 추가"

# 6. 로그인 기능 완료, main에 병합
git switch main
git merge feature/login
git branch -d feature/login
```

이 시나리오에서 특히 주목할 점은 3번 단계입니다. 로그인 기능을 개발하던 중 긴급 버그가 발생했을 때, 현재 작업 중인 `feature/login` 브랜치의 변경 사항을 커밋하거나 스태시한 후 `main` 브랜치로 전환합니다. 그리고 `main`에서 직접 수정하지 않고 `hotfix/critical-bug`라는 별도 브랜치를 만들어 버그를 수정합니다. 이렇게 함으로써 버그 수정 이력을 명확히 남기고, 문제가 발생했을 때 쉽게 롤백할 수 있습니다.

## 연습 문제

1. 다음 명령어를 순서대로 실행했을 때의 결과를 설명해보세요.
   - `git branch feature/update`
   - `git switch feature/update`
   - `git branch`

2. 현재 작업 중인 브랜치를 삭제하려고 하면 어떤 일이 발생하는지 설명하고, 올바른 삭제 방법을 서술해보세요.

3. 브랜치를 전환할 때 커밋하지 않은 변경 사항이 있어 전환이 거부되는 경우, 이를 해결하는 두 가지 방법을 설명해보세요.

---

📌 정답 및 해설

**문제 1 정답 및 해설:**

주어진 명령어를 순서대로 실행하면 다음과 같은 결과가 발생합니다. `git branch feature/update`는 현재 브랜치(main)와 동일한 위치에 `feature/update`라는 새 브랜치를 생성합니다. `git switch feature/update`는 현재 작업 중인 브랜치를 `feature/update`로 전환합니다. `git branch`는 현재 저장소의 모든 브랜치 목록을 표시하며, 현재 작업 중인 브랜치 앞에는 `*` 표시가 붙습니다. 따라서 출력 결과는 `* feature/update`와 `main` 두 줄이 표시되고, `feature/update` 앞에 `*`가 표시됩니다. 이때 `git branch` 명령어는 브랜치 생성 시점의 커밋과 아무런 관계없이, 단순히 모든 브랜치의 이름을 나열한다는 점에 주의해야 합니다.

**문제 2 정답 및 해설:**

현재 작업 중인 브랜치(HEAD가 가리키고 있는 브랜치)를 삭제하려고 하면 Git이 "error: Cannot delete branch '...' checked out at '...'"라는 오류 메시지를 출력하며 삭제를 거부합니다. 이는 자신이 현재 서 있는 발판을 스스로 제거할 수 없는 것과 같은 이치입니다. 올바른 삭제 방법은 먼저 다른 브랜치(main 등)로 전환한 후에 삭제 대상 브랜치를 삭제하는 것입니다. 예를 들어 `git switch main`으로 main 브랜치로 이동한 후, `git branch -d feature/update`로 feature/update 브랜치를 삭제합니다. 만약 삭제할 브랜치에 아직 병합되지 않은 커밋이 있다면 `-d` 대신 `-D`(강제 삭제) 옵션을 사용해야 하지만, 이 경우 커밋이 사라질 수 있으므로 신중해야 합니다.

**문제 3 정답 및 해설:**

브랜치 전환 시 커밋하지 않은 변경 사항이 있어 전환이 거부되는 경우, 두 가지 방법으로 해결할 수 있습니다. 첫 번째 방법은 변경 사항을 커밋하는 것입니다. `git add .`와 `git commit -m "임시 저장"`으로 현재 작업을 커밋한 후 브랜치를 전환합니다. 두 번째 방법은 `git stash` 명령어를 사용하여 변경 사항을 임시로 저장하는 것입니다. `git stash`는 작업 디렉토리의 변경 사항을 스택에 저장하고 작업 디렉토리를 깨끗한 상태로 만듭니다. 브랜치 전환 후 작업이 끝나면 다시 돌아와서 `git stash pop`으로 저장된 변경 사항을 복원할 수 있습니다. `git stash`는 아직 커밋할 준비가 되지 않은 작업을 임시로 보관할 때 특히 유용하며, 여러 번의 stash를 쌓아두고 선택적으로 복원할 수도 있습니다.
