# 변경 사항 확인: git diff

## 👨‍💻 실전 프로젝트: 코드 변경 추적하기

이번 실전 프로젝트에서는 `git diff`와 `git diff --staged`를 사용하여 코드의 변경 사항을 추적하는 방법을 직접 체험해보겠습니다. 파일을 수정하고, 스테이징한 후, 커밋하기 전에 각 단계에서 `git diff` 명령어를 실행하여 어떤 내용이 어떻게 바뀌었는지를 확인하는 과정을 반복합니다. 이는 실제 개발 현장에서 커밋을 만들기 전에 반드시 수행하는 습관으로, 불필요한 코드나 실수로 포함된 디버깅 코드를 걸러내는 데 큰 도움이 됩니다. 터미널을 열고 아래 명령어를 따라 입력하면서 diff 출력의 변화를 주의 깊게 관찰하시기 바랍니다.

```bash
# 1단계: 실습용 저장소를 생성하고 첫 번째 커밋을 만듭니다.
$ mkdir diff-practice && cd diff-practice && git init
$ echo "<h1>My Website</h1>" > index.html
$ git add index.html && git commit -m "Initial commit: add index.html"

# 2단계: 파일을 수정하고 git diff로 변경 사항을 확인합니다.
$ echo "<p>Welcome to my site</p>" >> index.html
$ git diff
diff --git a/index.html b/index.html
index a1b2c3d..d4e5f6f 100644
--- a/index.html
+++ b/index.html
@@ -1 +1,2 @@
 <h1>My Website</h1>
+<p>Welcome to my site</p>

# 3단계: 파일을 스테이징하고, git diff --staged로 스테이징된 변경 사항을 확인합니다.
$ git add index.html
$ git diff              # 출력 없음 (작업 디렉토리와 스테이징 영역이 동일하므로)
$ git diff --staged     # 스테이징된 변경 사항 출력!
diff --git a/index.html b/index.html
index a1b2c3d..d4e5f6f 100644
--- a/index.html
+++ b/index.html
@@ -1 +1,2 @@
 <h1>My Website</h1>
+<p>Welcome to my site</p>

# 4단계: 커밋하고 커밋 간 차이도 확인해봅니다.
$ git commit -m "Add welcome message"
$ git diff HEAD~1 HEAD
diff --git a/index.html b/index.html
...
```

여러분은 방금 `git diff` 명령어를 사용하여 코드 변경 사항을 추적하는 전 과정을 성공적으로 체험했습니다. 이처럼 `git diff`는 커밋하기 전에 변경 사항을 최종 검토하는 도구로 사용되며, `git diff --staged`는 곧 커밋될 내용을 미리 확인하는 용도로 사용됩니다. 이제 본격적으로 각 명령어의 상세 사용법을 알아보겠습니다.

---

개발을 하다 보면 "내가 지금까지 어떤 내용을 수정했는지"를 커밋하기 전에 꼼꼼히 확인해야 할 때가 있습니다. 특히 협업 프로젝트에서는 불필요한 변경이나 실수로 포함된 코드(예: 디버깅용 `console.log`, 주석 처리된 코드, 하드코딩된 비밀번호 등)가 없도록 확인하는 과정이 매우 중요합니다. 이러한 확인 과정을 생략하면 나중에 불필요한 코드가 배포 환경에 포함되는 사고로 이어질 수 있습니다. `git diff`는 파일의 변경 사항을 **줄 단위**로 비교하여 보여주는 명령어로, Git에서 제공하는 가장 강력한 차이 비교 도구입니다. 이 장에서는 커밋하기 전에 무엇이 바뀌었는지 확인하는 방법을 집중적으로 알아보겠습니다.

## 학습 목표

- `git diff` 명령어로 작업 디렉토리와 스테이징 영역 간의 차이를 확인할 수 있습니다
- `git diff --staged` 명령어로 스테이징된 변경 사항을 확인할 수 있습니다
- 두 커밋 또는 브랜치 간의 차이를 비교할 수 있습니다
- diff 출력 구조를 해석하고 이해할 수 있습니다

이 목표들을 달성하면 Git에서 변경 사항을 비교하고 분석하는 능력을 갖추게 됩니다. 이는 코드 리뷰, 디버깅, 충돌 해결 등 다양한 상황에서 핵심적으로 활용되는 기술입니다.

## 1. 작업 디렉토리 vs Staging Area: `git diff`

아직 `git add`로 스테이징하지 않은 변경 사항, 즉 작업 디렉토리에서 수정되었지만 아직 스테이징 영역에 추가되지 않은 변경 사항을 확인할 때 `git diff` 명령어를 사용합니다. 이 명령어는 스테이징 영역에 있는 파일의 내용과 작업 디렉토리에 있는 파일의 내용을 비교하여 그 차이를 출력합니다. `git add`를 실행하기 전에 최종적으로 변경 사항을 검토하는 용도로 가장 많이 사용됩니다.

```bash
# 모든 파일의 변경 사항 확인
$ git diff

# 특정 파일만 확인
$ git diff index.html
```

**출력 예시:**

```diff
$ git diff
diff --git a/index.html b/index.html
index a1b2c3d..d4e5f6g 100644
--- a/index.html
+++ b/index.html
@@ -10,6 +10,7 @@
     <title>내 웹사이트</title>
   </head>
   <body>
+    <h1>환영합니다!</h1>      ← 초록색: 추가된 줄
     <p>내용입니다.</p>
-    <p>이전 텍스트</p>        ← 빨간색: 삭제된 줄
+    <p>새로운 텍스트</p>      ← 수정됨 (삭제 + 추가)
   </body>
 </html>
```

diff 출력 구조를 자세히 살펴보겠습니다:

```
@@ -10,6 +10,7 @@
```

- `-10,6`: 원본 파일의 10번째 줄부터 6줄을 의미합니다.
- `+10,7`: 변경 후 파일의 10번째 줄부터 7줄을 의미합니다.
- `-`로 시작하는 줄: 삭제된 내용(빨간색)입니다.
- `+`로 시작하는 줄: 추가된 내용(초록색)입니다.

`@@ -10,6 +10,7 @@`는 원본에서는 10번째 줄부터 6줄이 있었는데, 변경 후에는 10번째 줄부터 7줄이 되었다는 것을 의미합니다. 즉, 1줄이 추가되었다는 뜻입니다. Git의 diff는 변경된 위치를 이렇게 청크(chunk) 단위로 표시하여, 대용량 파일에서도 변경된 부분을 쉽게 찾을 수 있도록 도와줍니다.

## 2. Staging Area vs 저장소: `git diff --staged`

`git add`로 스테이징된 변경 사항, 즉 다음 커밋에 포함될 내용을 확인할 때는 `--staged` 옵션을 사용합니다. 이는 "내가 지금 커밋하려는 내용이 정확히 무엇인지"를 최종 확인하는 용도로 사용됩니다. `git commit`을 실행하기 직전에 `git diff --staged`를 실행하여 커밋될 내용을 검토하는 습관을 들이면, 불필요한 파일이나 실수로 포함된 코드를 커밋하는 실수를 방지할 수 있습니다.

```bash
git diff --staged
# 또는
git diff --cached
```

`--cached`는 `--staged`와 완전히 동일한 옵션입니다. Git 초기 버전에서 사용되던 용어이며, 현재는 `--staged`가 더 직관적이므로 주로 `--staged`를 사용합니다.

```bash
$ echo "<footer>ⓒ 2024</footer>" >> index.html
$ git add index.html

$ git diff                    # 출력 없음 (이미 Staging에 있음)
$ git diff --staged           # 변경 사항 출력!
diff --git a/index.html b/index.html
+    <footer>ⓒ 2024</footer>
```

위 예제에서 `git add` 후 `git diff`를 실행하면 아무것도 출력되지 않습니다. 왜냐하면 `git diff`는 작업 디렉토리와 스테이징 영역을 비교하는데, `git add`로 인해 스테이징 영역이 작업 디렉토리와 동일한 내용을 가지게 되었기 때문입니다. 이때 `git diff --staged`를 실행하면 스테이징 영역과 마지막 커밋 간의 차이, 즉 곧 커밋될 내용이 출력됩니다.

## 3. 커밋 간 비교

저장소에 기록된 여러 커밋 간의 차이를 비교할 수도 있습니다. 이 기능은 "이번 릴리스에서 어떤 변경 사항이 있었는지" 또는 "특정 버그가 언제引入되었는지"를 추적할 때 매우 유용합니다.

```bash
# 현재 HEAD와 비교
$ git diff HEAD

# 특정 커밋과 비교
$ git diff a1b2c3d

# 두 커밋 사이의 변경 사항
$ git diff a1b2c3d..d4e5f6f

# 두 브랜치 비교
$ git diff main..feature/login
```

`git diff HEAD`는 현재 작업 디렉토리와 가장 최근 커밋(HEAD)을 비교합니다. 이는 스테이징 여부와 관계없이 마지막 커밋 이후의 모든 변경 사항을 보여줍니다. 두 커밋 간의 비교(`git diff a1b2c3d..d4e5f6f`)는 특정 기간 동안의 전체 변경 사항을 한눈에 파악하고 싶을 때 사용합니다. 브랜치 간 비교(`git diff main..feature/login`)는 병합(merge) 전에 두 브랜치 간의 차이를 미리 확인하여 충돌 가능성을 예측하는 데 유용합니다.

**커밋 비교 예시:**

```bash
$ git log --oneline
d4e5f6f (HEAD) 푸터 추가
a1b2c3d 첫 번째 커밋

# 첫 번째 커밋과 두 번째 커밋의 차이
$ git diff a1b2c3d..d4e5f6f
diff --git a/index.html b/index.html
+    <footer>ⓒ 2024</footer>
```

## 4. 요약 보기: `git diff --stat`

변경된 파일과 줄 수만 간략히 표시하고 싶을 때는 `--stat` 옵션을 사용합니다. 이 옵션은 실제 변경 내용의 상세 diff는 생략하고, 각 파일에서 몇 줄이 추가되고 삭제되었는지만 요약하여 보여줍니다. 전체 변경 사항의 규모를 빠르게 파악하고 싶을 때 유용합니다.

```bash
$ git diff --stat
 index.html | 3 ++-
 style.css  | 5 +++++
 README.md  | 1 +
 3 files changed, 8 insertions(+), 1 deletion(-)
```

`--stat` 출력에서 `3 ++-`는 index.html 파일에 2줄이 추가되고 1줄이 삭제되었음을 의미합니다. 맨 아래 줄에는 총 변경된 파일 수, 추가된 줄 수, 삭제된 줄 수가 요약되어 표시됩니다.

## 5. diff 도구 사용하기

복잡한 diff는 터미널에서 보는 것보다 GUI 도구로 보는 것이 더 편리할 때가 많습니다. Git은 외부 diff 도구와 연동하여 사용할 수 있는 `git difftool` 명령어를 제공합니다.

```bash
git difftool
```

설정에 따라 VS Code, Beyond Compare, Kaleidoscope 등 다양한 외부 도구가 실행됩니다. GUI 도구는 터미널보다 더 직관적으로 변경 사항을 시각화해주므로, 대규모 변경 사항을 검토할 때 특히 유용합니다. `git difftool --tool-help` 명령어로 사용 가능한 diff 도구 목록을 확인할 수 있습니다.

## 6. 실습 시나리오

지금까지 배운 내용을 하나의 실습 시나리오로 따라 해보겠습니다. 각 단계에서 `git diff`와 `git diff --staged`의 출력 차이를 주의 깊게 확인하시기 바랍니다. 특히 `git add`를 실행하기 전후로 `git diff`의 출력이 어떻게 달라지는지 관찰하는 것이 중요합니다.

```bash
# 1. 파일 수정
$ echo "새로운 내용" > hello.txt

# 2. 변경 사항 확인 (Unstaged)
$ git diff
diff --git a/hello.txt b/hello.txt
+새로운 내용

# 3. 스테이징
$ git add hello.txt
$ git diff              # 출력 없음

# 4. 스테이징된 변경 사항 확인
$ git diff --staged
diff --git a/hello.txt b/hello.txt
+새로운 내용

# 5. 커밋
$ git commit -m "hello.txt 추가"

# 6. 커밋 후에는 diff 출력 없음
$ git diff              # 출력 없음
$ git diff --staged     # 출력 없음

# 7. 이전 커밋과 비교하려면
$ git diff HEAD~1 HEAD
```

2번 단계에서 `git diff`가 변경 사항을 보여주지만, 3번 단계에서 `git add`를 실행한 후에는 더 이상 `git diff`가 출력되지 않습니다. 이는 작업 디렉토리와 스테이징 영역이 동일한 내용을 가지게 되었기 때문입니다. 대신 4번 단계에서 `git diff --staged`가 스테이징된 변경 사항을 보여줍니다. 5번 단계에서 커밋한 후에는 `git diff`와 `git diff --staged` 모두 출력되지 않으며, 이전 커밋과 비교하려면 `git diff HEAD~1 HEAD`를 사용해야 합니다.

## 한눈에 정리

| 명령어 | 비교 대상 | 사용 시점 |
|--------|----------|----------|
| `git diff` | 작업 디렉토리 ↔ 스테이징 영역 | `git add` 하기 전, 수정 사항 최종 확인 |
| `git diff --staged` | 스테이징 영역 ↔ 저장소 | `git commit` 하기 전, 커밋될 내용 최종 확인 |
| `git diff <커밋A>..<커밋B>` | 두 커밋 간 전체 차이 | 특정 기간 동안의 변경 사항 분석 |
| `git diff <브랜치A>..<브랜치B>` | 두 브랜치 간 차이 | 브랜치 병합 전 충돌 예측 |

위 표는 `git diff` 명령어의 네 가지 주요 사용 패턴을 정리합니다. 각 패턴은 비교 대상과 사용 시점이 다르므로, 상황에 맞게 적절한 명령어를 선택하는 것이 중요합니다. `git diff`는 커밋 전 검토 도구로, `git diff --staged`는 커밋 직전 최종 확인 도구로 기억하면 됩니다.
