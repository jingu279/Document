# 변경 사항 확인: git diff

개발을 하다 보면 "내가 지금까지 어떤 내용을 수정했는지"를 커밋하기 전에 꼼꼼히 확인해야 할 때가 있습니다. 특히 협업 프로젝트에서는 불필요한 변경이나 실수로 포함된 코드가 없도록 확인하는 과정이 매우 중요합니다. `git diff`는 파일의 변경 사항을 **줄 단위**로 비교하여 보여주는 명령어입니다. 이 장에서는 커밋하기 전에 무엇이 바뀌었는지 확인하는 방법을 알아보겠습니다.


## 학습 목표

- `git diff` 명령어로 작업 디렉토리와 스테이징 영역 간의 차이를 확인할 수 있습니다
- `git diff --staged` 명령어로 스테이징된 변경 사항을 확인할 수 있습니다
- 두 커밋 또는 브랜치 간의 차이를 비교할 수 있습니다
- diff 출력 구조를 해석하고 이해할 수 있습니다

## 1. 작업 디렉토리 vs Staging Area: `git diff`

아직 `git add`하지 않은 변경 사항을 확인할 때 `git diff` 명령어를 사용합니다.

```bash
# 모든 파일의 변경 사항 확인
$ git diff

# 특정 파일만 확인
$ git diff index.html
```

**출력 예시:**

```diff
$ git diff
diff --git a/index.html b/index.html
index a1b2c3d..d4e5f6g 100644
--- a/index.html
+++ b/index.html
@@ -10,6 +10,7 @@
     <title>내 웹사이트</title>
   </head>
   <body>
+    <h1>환영합니다!</h1>      ← 초록색: 추가된 줄
     <p>내용입니다.</p>
-    <p>이전 텍스트</p>        ← 빨간색: 삭제된 줄
+    <p>새로운 텍스트</p>      ← 수정됨 (삭제 + 추가)
   </body>
 </html>
```

**diff 출력 구조:**

```
@@ -10,6 +10,7 @@
```

- `-10,6`: 원본 파일의 10번째 줄부터 6줄
- `+10,7`: 변경 후 파일의 10번째 줄부터 7줄
- `-`: 삭제된 줄 (빨간색)
- `+`: 추가된 줄 (초록색)

## 2. Staging Area vs 저장소: `git diff --staged`

`git add`로 스테이징된 변경 사항을 확인할 때는 `--staged` 옵션을 사용합니다.

```bash
git diff --staged
# 또는
git diff --cached
```

```bash
$ echo "<footer>ⓒ 2024</footer>" >> index.html
$ git add index.html

$ git diff                    # 출력 없음 (이미 Staging에 있음)
$ git diff --staged           # 변경 사항 출력!
diff --git a/index.html b/index.html
+    <footer>ⓒ 2024</footer>
```

## 3. 커밋 간 비교

저장소에 기록된 여러 커밋 간의 차이를 비교할 수도 있습니다.

```bash
# 현재 HEAD와 비교
$ git diff HEAD

# 특정 커밋과 비교
$ git diff a1b2c3d

# 두 커밋 사이의 변경 사항
$ git diff a1b2c3d..d4e5f6f

# 두 브랜치 비교
$ git diff main..feature/login
```

**커밋 비교 예시:**

```bash
$ git log --oneline
d4e5f6f (HEAD) 푸터 추가
a1b2c3d 첫 번째 커밋

# 첫 번째 커밋과 두 번째 커밋의 차이
$ git diff a1b2c3d..d4e5f6f
diff --git a/index.html b/index.html
+    <footer>ⓒ 2024</footer>
```

## 4. 요약 보기: `git diff --stat`

변경된 파일과 줄 수만 간략히 표시하고 싶을 때는 `--stat` 옵션을 사용합니다.

```bash
$ git diff --stat
 index.html | 3 ++-
 style.css  | 5 +++++
 README.md  | 1 +
 3 files changed, 8 insertions(+), 1 deletion(-)
```

## 5. diff 도구 사용하기

복잡한 diff는 터미널 대신 GUI 도구로 보는 것이 편리합니다.

```bash
git difftool
```

설정에 따라 VS Code, Beyond Compare 등 외부 도구가 실행됩니다.

## 6. 실습 시나리오

지금까지 배운 내용을 하나의 실습 시나리오로 따라 해보겠습니다. 각 단계에서 `git diff`와 `git diff --staged`의 출력 차이를 주의 깊게 확인하시기 바랍니다.

```bash
# 1. 파일 수정
$ echo "새로운 내용" > hello.txt

# 2. 변경 사항 확인 (Unstaged)
$ git diff
diff --git a/hello.txt b/hello.txt
+새로운 내용

# 3. 스테이징
$ git add hello.txt
$ git diff              # 출력 없음

# 4. 스테이징된 변경 사항 확인
$ git diff --staged
diff --git a/hello.txt b/hello.txt
+새로운 내용

# 5. 커밋
$ git commit -m "hello.txt 추가"

# 6. 커밋 후에는 diff 출력 없음
$ git diff              # 출력 없음
$ git diff --staged     # 출력 없음

# 7. 이전 커밋과 비교하려면
$ git diff HEAD~1 HEAD
```

## 한눈에 정리

| 명령어 | 비교 대상 | 사용 시점 |
|--------|----------|----------|
| `git diff` | 작업 디렉토리 ↔ 스테이징 영역 | `git add` 하기 전, 수정 사항 최종 확인 |
| `git diff --staged` | 스테이징 영역 ↔ 저장소 | `git commit` 하기 전, 커밋될 내용 최종 확인 |
| `git diff <커밋A>..<커밋B>` | 두 커밋 간 전체 차이 | 특정 기간 동안의 변경 사항 분석 |
| `git diff <브랜치A>..<브랜치B>` | 두 브랜치 간 차이 | 브랜치 병합 전 충돌 예측 |

## 연습 문제

1. `git diff`와 `git diff --staged`의 차이점을 서술하고, 각각 언제 사용하는지 설명하시오.
2. diff 출력에서 `@@ -10,6 +10,7 @@`는 무엇을 의미하는지 설명하시오.
3. `git diff a1b2c3d..d4e5f6f` 명령어를 실행했을 때 아무 출력도 나오지 않았다면 어떤 상황임을 의미하는지 서술하시오.
