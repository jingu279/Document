# 커밋 기록 확인

## 👨‍💻 실전 프로젝트: 커밋 히스토리 탐험하기

이번 실전 프로젝트에서는 `git log` 명령어의 다양한 옵션을 활용하여 커밋 히스토리를 탐험하는 방법을 체험해보겠습니다. 먼저 여러 개의 커밋이 있는 저장소를 준비한 후, 각 옵션별로 로그 출력 결과가 어떻게 달라지는지 직접 확인합니다. 히스토리 탐험 능력은 협업 프로젝트에서 코드의 변화를 추적하고, 특정 변경 사항을 찾는 데 필수적인 기술입니다. 터미널을 열고 아래 명령어를 따라 입력하면서 각 옵션의 출력 차이를 주의 깊게 관찰하시기 바랍니다.

```bash
# 1단계: 실습용 저장소를 생성하고 여러 개의 커밋을 만듭니다.
$ mkdir history-lab && cd history-lab && git init
$ echo "# Project" > README.md && git add . && git commit -m "Initial commit"
$ echo "console.log('hello');" > app.js && git add . && git commit -m "Add app.js"
$ echo "body { color: blue; }" > style.css && git add . && git commit -m "Add CSS styles"
$ echo "console.log('world');" >> app.js && git add . && git commit -m "Update app.js with world log"

# 2단계: 기본 로그를 확인합니다 (전체 커밋 히스토리).
$ git log

# 3단계: --oneline 옵션으로 각 커밋을 한 줄로 간략하게 표시합니다.
$ git log --oneline
a1b2c3d Update app.js with world log
d4e5f6f Add CSS styles
g7h8i9j Add app.js
k1l2m3n Initial commit

# 4단계: --oneline --graph --all 옵션으로 브랜치 구조를 시각화합니다.
$ git log --oneline --graph --all
* a1b2c3d (HEAD -> main) Update app.js with world log
* d4e5f6f Add CSS styles
* g7h8i9j Add app.js
* k1l2m3n Initial commit

# 5단계: 최근 2개의 커밋만 표시합니다.
$ git log -2
commit a1b2c3d (HEAD -> main)
    Update app.js with world log

commit d4e5f6f
    Add CSS styles

# 6단계: --author 옵션으로 특정 작성자의 커밋을 필터링합니다.
$ git log --author="Your Name"

# 7단계: --since 옵션으로 특정 기간 이후의 커밋을 조회합니다.
$ git log --since="2026-01-01"
```

여러분은 방금 `git log`의 다양한 옵션을 사용하여 커밋 히스토리를 효과적으로 탐색하는 방법을 익혔습니다. 각 옵션은 서로 다른 상황에서 유용하게 사용되며, 실무에서는 이들을 조합하여 사용하는 경우가 많습니다. 이제 본격적으로 각 명령어와 옵션의 상세 사용법을 알아보겠습니다.

---

프로젝트를 진행하다 보면 "이전에 어떤 변경 사항이 있었는지", "이 코드는 누가, 언제 작성했는지"를 확인해야 하는 상황이 자주 발생합니다. 예를 들어, 갑자기 발생한 버그의 원인을 찾기 위해 최근 변경 사항을 추적하거나, 특정 기능이 언제 추가되었는지 확인해야 할 때가 있습니다. Git의 강력한 기능 중 하나는 바로 이러한 프로젝트의 변경 이력을 자유롭게 살펴볼 수 있다는 것입니다. 이 장에서는 `git log`, `git diff`, `git show` 명령어를 사용하여 커밋 기록을 확인하고 변경 사항을 비교하는 방법을 알아보겠습니다.

## 학습 목표

- `git log` 명령어로 커밋 기록을 조회할 수 있습니다
- 다양한 `git log` 옵션을 활용하여 원하는 형태로 이력을 확인할 수 있습니다
- `git diff` 명령어로 변경 사항을 비교할 수 있습니다
- `git show` 명령어로 특정 커밋의 상세 내용을 확인할 수 있습니다

이 목표들을 달성하면 프로젝트의 전체 이력을 자유자재로 탐색하고, 필요한 정보를 신속하게 찾아낼 수 있는 능력을 갖추게 됩니다. 이는 디버깅, 코드 리뷰, 협업 등 다양한 상황에서 매우 유용하게 활용됩니다.

## 1. 기본 로그 확인: `git log`

가장 기본적인 형태의 `git log` 명령어는 전체 커밋 기록을 최신순으로 보여줍니다. 즉, 가장 최근에 만든 커밋이 가장 위에 표시되며, 시간순으로 과거의 커밋들이 아래로 이어집니다. `git log`는 페이지 단위로 출력되며, 스페이스바를 누르면 다음 페이지로, `q`를 누르면 로그 화면에서 빠져나올 수 있습니다.

```bash
git log
```

**출력 예시:**
```
commit 7a3fb42e8d9c1f5b2a4c6d8e0f1a2b3c4d5e6f7a (HEAD -> main)
Author: Your Name <your.email@example.com>
Date:   Mon Jul 10 14:30:00 2026 +0900

    index.html 스타일 업데이트 및 new-style.css 추가

commit 1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b
Author: Your Name <your.email@example.com>
Date:   Mon Jul 10 12:00:00 2026 +0900

    첫 번째 커밋: 프로젝트 초기화
```

각 커밋은 다음 정보를 포함합니다:

- **커밋 해시 (Commit Hash):** `7a3fb42...`와 같은 고유한 40자리 16진수 식별자로, 전 세계에서 유일무이한 값입니다.
- **Author:** 커밋을 작성한 사람의 이름과 이메일 주소입니다.
- **Date:** 커밋이 생성된 날짜와 시간입니다.
- **Commit Message:** 커밋할 때 작성한 메시지로, 변경 사항에 대한 설명을 담고 있습니다.

**커밋 해시의 의미:**

커밋 해시는 SHA-1 알고리즘으로 생성된 고유 식별자입니다. 이 해시는 다음과 같은 정보를 기반으로 만들어집니다:

- 파일 스냅샷의 내용
- 부모 커밋의 해시
- 커밋 시간
- 작성자 정보

따라서 **같은 내용이라도 다른 시간에 커밋하면 다른 해시**가 생성됩니다. 이는 각 커밋이 전 세계에서 유일함을 보장하며, 커밋 해시를 통해 특정 커밋을 정확하게 참조할 수 있습니다. 커밋 해시는 보통 앞 7자리만 사용해도 충분히 고유하므로, 실무에서는 짧은 해시를 주로 사용합니다.

```bash
$ git rev-parse HEAD
7a3fb42e8d9c1f5b2a4c6d8e0f1a2b3c4d5e6f7a

# 앞 7자리만 사용해도 충분히 고유함
$ git show 7a3fb42  # 가능!
```

`git rev-parse HEAD` 명령어는 현재 HEAD가 가리키는 커밋의 전체 해시를 출력합니다. `HEAD`는 현재 체크아웃된 커밋을 가리키는 포인터로, 대부분의 Git 명령어에서 기본 참조 대상으로 사용됩니다.

## 2. 다양한 로그 옵션

`git log`는 다양한 옵션을 제공하여 원하는 형태로 이력을 조회할 수 있습니다. 상황에 맞는 옵션을 선택하여 사용하면, 복잡한 히스토리에서도 필요한 정보를 신속하게 찾을 수 있습니다. 아래에서 가장 유용한 옵션들을 하나씩 살펴보겠습니다.

### 한 줄로 간략하게 보기
```bash
git log --oneline
```
```
7a3fb42 (HEAD -> main) index.html 스타일 업데이트 및 new-style.css 추가
1a2b3c4 첫 번째 커밋: 프로젝트 초기화
```

`--oneline` 옵션은 각 커밋을 짧은 해시(7자리)와 커밋 메시지 제목만으로 한 줄에 표시합니다. 커밋 수가 많을 때 가장 자주 사용되는 옵션으로, 전체 히스토리를 한눈에 파악할 수 있습니다.

### 그래프 형태로 보기 (브랜치 구조 확인)
```bash
git log --oneline --graph --all
```
```
* 7a3fb42 (HEAD -> main) index.html 스타일 업데이트 및 new-style.css 추가
* 1a2b3c4 첫 번째 커밋: 프로젝트 초기화
```

`--graph` 옵션은 브랜치의 분기와 병합을 그래프 형태로 시각화하여 보여줍니다. `--all` 옵션을 함께 사용하면 모든 브랜치의 커밋을 한꺼번에 볼 수 있어, 프로젝트의 전체적인 브랜치 구조를 파악하는 데 매우 유용합니다.

### 최근 N개의 커밋만 보기
```bash
git log -3          # 최근 3개의 커밋만 표시
git log -n 5        # 최근 5개의 커밋만 표시
```

커밋 수가 많은 프로젝트에서는 전체 로그를 보는 것보다 최근 N개의 커밋만 확인하는 것이 효율적입니다. `-<숫자>` 또는 `-n <숫자>` 옵션을 사용하면 출력할 커밋의 개수를 제한할 수 있습니다.

### 특정 작성자의 커밋만 보기
```bash
git log --author="홍길동"
```

팀 프로젝트에서 특정 팀원의 작업 내역만 확인해야 할 때 유용합니다. `--author` 옵션은 이름이나 이메일의 일부만 입력해도 일치하는 커밋을 찾아줍니다.

### 날짜/시간 범위로 검색
```bash
git log --since="2025-01-01" --until="2025-12-31"
git log --after="2 weeks ago"   # 2주 전 이후 커밋
git log --before="yesterday"    # 어제 이전 커밋
```

날짜 기반 검색은 특정 기간 동안의 변경 사항을 분석할 때 매우 유용합니다. `--since`와 `--after`는 같은 기능이며, `--until`과 `--before`도 같은 기능입니다. 날짜는 "2025-01-01"과 같은 절대 날짜뿐만 아니라 "2 weeks ago"와 같은 상대적인 표현도 사용할 수 있습니다.

### 특정 문자열이 포함된 커밋 찾기
```bash
git log --grep="버그"           # 메시지에 "버그"가 포함된 커밋
git log --grep="fix" -i         # 대소문자 무시하고 "fix" 검색
git log -S "console.log"        # 코드에 "console.log"가 추가/제거된 커밋
```

`--grep`은 커밋 메시지에서 특정 문자열을 검색하고, `-S`(pickaxe)는 실제 코드 변경 내용에서 특정 문자열이 추가되거나 삭제된 커밋을 찾습니다. 예를 들어, `git log -S "console.log"`는 "console.log"라는 문자열이 코드에 추가되거나 삭제된 모든 커밋을 찾아줍니다.

### 로그 출력 형식 커스터마이징
```bash
git log --pretty=format:"%h - %an, %ar : %s"
# a1b2c3d - 홍길동, 2 days ago : README 업데이트
# d4e5f6f - 김철수, 5 days ago : 로그인 버그 수정

# %h : 짧은 해시
# %an : 작성자 이름
# %ar : 상대적 시간 (2 days ago)
# %s : 커밋 메시지 제목
# %d : ref 이름 (브랜치, 태그)
```

`--pretty=format` 옵션을 사용하면 로그 출력 형식을 원하는 대로 완전히 커스터마이징할 수 있습니다. 이 옵션은 스크립트에서 Git 로그를 파싱해야 할 때나, 팀에서 일관된 로그 형식을 사용하고자 할 때 특히 유용합니다.

### 특정 파일의 변경 이력만 보기
```bash
git log -- index.html
```

파일 경로를 인자로 전달하면 해당 파일의 변경 이력만 필터링하여 보여줍니다. 예를 들어, 특정 파일이 언제 생성되었고, 어떤 커밋에서 수정되었는지를 추적할 수 있습니다.

### 변경된 파일 목록만 확인
```bash
git log --name-only
git log --name-status    # 변경 유형까지 표시
```

`--name-only`는 각 커밋에서 변경된 파일의 이름만 표시하고, `--name-status`는 각 파일의 변경 유형(Added, Modified, Deleted)까지 함께 표시합니다. 커밋의 범위를 빠르게 파악하고 싶을 때 유용합니다.

## 3. 변경 사항 비교하기: `git diff`

커밋 기록을 확인하는 것만으로는 각 커밋에서 어떤 내용이 실제로 변경되었는지 구체적으로 알기 어려울 수 있습니다. 이때 `git diff` 명령어를 사용하면 파일의 내용이 어떻게 바뀌었는지 줄 단위로 정확히 확인할 수 있습니다. `git diff`는 Git에서 가장 강력한 명령어 중 하나이며, 코드 리뷰와 디버깅에 필수적으로 사용됩니다.

이 명령어에 대한 자세한 내용은 다음 장인 "05_변경_사항_확인.md"에서 집중적으로 다루므로, 여기서는 기본적인 개념과 사용법만 간략히 소개합니다.

### 스테이징되지 않은 변경 사항 확인 (Working Directory vs Staging Area)
```bash
git diff
```

### 스테이징된 변경 사항 확인 (Staging Area vs Repository)
```bash
git diff --staged
```

### 두 커밋 간의 차이 확인
```bash
git diff <커밋해시1> <커밋해시2>
```

### 실제 diff 출력 예시 (커밋 간 비교)
```bash
$ git diff a1b2c3d..d4e5f6f
diff --git a/README.md b/README.md
index e69de29..d95f3ad 100644
--- a/README.md
+++ b/README.md
@@ -0,0 +1,3 @@
+# My Project
+
+This is a sample project for learning Git.
```

**diff 출력 읽는 법:**
- `--- a/file` : 원본 파일 (a)
- `+++ b/file` : 비교 대상 파일 (b)
- `@@ -시작줄,줄수 +시작줄,줄수 @@` : 변경 위치
- `-` 로 시작하는 줄: 삭제된 내용
- `+` 로 시작하는 줄: 추가된 내용

### 특정 파일의 변경 이력만 diff
```bash
git diff a1b2c3d..d4e5f6f -- README.md
```

### 단어 단위로 diff 보기
```bash
git diff --word-diff
```

`--word-diff` 옵션은 줄 단위가 아니라 단어 단위로 변경 사항을 표시합니다. 긴 줄의 일부만 수정되었을 때, 어떤 단어가 변경되었는지 더 정확하게 파악할 수 있습니다.

## 4. 특정 커밋의 상세 내용 확인: `git show`

`git show` 명령어는 특정 커밋의 변경 사항과 메타데이터를 자세히 보여줍니다. `git log`가 커밋의 목록을 보여주는 반면, `git show`는 하나의 커밋에 집중하여 해당 커밋의 전체 변경 내용을 diff 형식으로 출력합니다. 인자 없이 `git show`를 실행하면 가장 최근 커밋(HEAD)의 내용을 보여줍니다.

```bash
git show 7a3fb42
```

**출력 예시:**
```bash
$ git show d4e5f6f
commit d4e5f6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d
Author: 홍길동 <hong@example.com>
Date:   Tue Jul 11 10:30:00 2026 +0900

    README에 설치 방법 추가

diff --git a/README.md b/README.md
index d95f3ad..b1c2d3e 100644
--- a/README.md
+++ b/README.md
@@ -1,3 +1,8 @@
 # My Project

 This is a sample project for learning Git.
+
+## Installation
+
+```bash
+npm install my-project
+```
```

`git show`의 출력은 커밋 메타데이터(해시, 작성자, 날짜, 메시지)와 해당 커밋의 diff를 함께 보여줍니다. 이를 통해 하나의 커밋에서 어떤 파일이 어떻게 변경되었는지를 한눈에 파악할 수 있습니다. 코드 리뷰에서 특정 커밋의 변경 사항을 확인할 때 가장 자주 사용되는 명령어입니다.

## 5. git log와 git diff 종합 예제

지금까지 배운 명령어들을 종합적으로 활용하는 예제를 살펴보겠습니다. 실제 개발 현장에서는 이 명령어들을 단독으로 사용하기보다는, 상황에 맞게 조합하여 사용하는 경우가 많습니다. 아래 예제를 통해 각 명령어의 실용적인 사용법을 익혀보시기 바랍니다.

```bash
# 1. 최근 5개 커밋을 한 줄로 확인
$ git log --oneline -5
a1b2c3d README에 설치 방법 추가
d4e5f6f 로그인 페이지 구현
g7h8i9j 메인 페이지 레이아웃 수정
k1l2m3n 첫 번째 커밋: 프로젝트 초기화

# 2. 특정 범위의 변경 사항 확인
$ git diff g7h8i9j..a1b2c3d

# 3. 특정 파일이 어떤 커밋에서 변경되었는지
$ git log --oneline -- README.md
a1b2c3d README에 설치 방법 추가
k1l2m3n 첫 번째 커밋: 프로젝트 초기화

# 4. 각 커밋별로 변경된 파일 목록
$ git log --oneline --stat
a1b2c3d README에 설치 방법 추가
 README.md | 5 +++++
 1 file changed, 5 insertions(+)

d4e5f6f 로그인 페이지 구현
 login.html | 20 ++++++++++++++++++++
 style.css  | 10 ++++++++++
 2 files changed, 30 insertions(+)
```

이러한 명령어들을 활용하면 프로젝트의 이력을 쉽게 추적하고, 어떤 변경이 언제, 왜 발생했는지 명확하게 파악할 수 있습니다. 예를 들어, 새로운 버그가 발견되었을 때 `git log --oneline -10`으로 최근 변경 사항을 빠르게 훑어보고, 의심되는 커밋에 대해 `git show <해시>`로 상세 내용을 확인하는 방식으로 문제를 진단할 수 있습니다.

## 한눈에 정리

| 명령어 | 목적 | 주요 옵션 |
|--------|------|----------|
| `git log` | 커밋 기록 조회 | `--oneline` (간략히), `--graph` (그래프 형태), `--author` (작성자별), `--grep` (메시지 검색) |
| `git diff` | 변경 사항 비교 | `--staged` (스테이징된 변경), `--stat` (요약), `--word-diff` (단어 단위) |
| `git show` | 특정 커밋 상세 확인 | `<커밋해시>` (특정 커밋 지정) |

위 표는 이 장에서 배운 세 가지 주요 명령어를 요약합니다. `git log`는 "어떤 커밋이 있는지", `git diff`는 "무엇이 바뀌었는지", `git show`는 "이 커밋의 자세한 내용은 무엇인지"를 확인할 때 각각 사용합니다.

## 연습 문제

1. `git log --oneline --graph --all` 명령어는 어떤 상황에서 유용한지 서술하시오.
2. 커밋 해시(Commit Hash)는 어떤 정보를 기반으로 생성되는지 세 가지 이상 쓰시오.
3. `git diff`와 `git diff --staged`의 차이점을 설명하고, 각각 언제 사용하는지 서술하시오.

📌 정답 및 해설

**문제 1 정답 및 해설:**

`git log --oneline --graph --all` 명령어는 모든 브랜치의 커밋 히스토리를 한 줄짜리 메시지와 그래프 형태로 시각화하여 보여줍니다. 이 명령어는 특히 여러 브랜치가 복잡하게 얽힌 상황에서 유용합니다. 예를 들어 feature 브랜치에서 작업한 내용을 main 브랜치에 병합한 후, 각 브랜치의 커밋이 어떤 순서로 연결되어 있는지 한눈에 파악할 때 사용합니다. `--graph` 옵션은 브랜치의 분기와 병합을 선으로 표시하여 시각적 이해를 돕고, `--all` 옵션은 현재 브랜치뿐만 아니라 모든 브랜치의 히스토리를 표시합니다. 병합(Merge)이 발생한 후 브랜치 구조를 팀원들과 공유하거나, 커밋 히스토리를 발표 자료로 사용할 때 특히 유용합니다.

**문제 2 정답 및 해설:**

커밋 해시(Commit Hash)는 다음 세 가지 이상의 정보를 기반으로 SHA-1 해시 함수를 통해 생성됩니다. 첫째, 커밋에 포함된 파일의 스냅샷 내용입니다. 모든 파일의 내용과 디렉토리 구조가 트리 객체로 저장되며, 이 트리 객체의 해시가 커밋 해시 계산에 포함됩니다. 둘째, 직전 커밋(Parent Commit)의 해시 값입니다. 이전 커밋을 가리키는 포인터 정보가 포함되어 커밋들이 체인처럼 연결됩니다. 셋째, 커밋 메시지와 작성자 정보(이름, 이메일), 커밋 시간의 타임스탬프입니다. 또한 committer 정보와 같은 추가 메타데이터도 해시 계산에 포함되므로, 이러한 요소 중 하나라도 변경되면 완전히 다른 해시 값이 생성됩니다.

**문제 3 정답 및 해설:**

`git diff`는 작업 디렉토리에서 수정되었지만 아직 스테이징되지 않은 변경 사항(Modified ↔ Unmodified)을 보여줍니다. 개발자가 파일을 수정한 후 `git add`를 실행하기 전에 어떤 내용이 바뀌었는지 확인할 때 사용합니다. 반면 `git diff --staged`는 스테이징 영역에 추가되었지만 아직 커밋되지 않은 변경 사항(Staged ↔ 직전 커밋)을 보여줍니다. `git add`를 실행한 후 `git commit`을 실행하기 전에 "내가 스테이징한 내용이 정확한가"를 최종 검토할 때 사용합니다. 간단히 말해, `git diff`는 "수정한 내용이 무엇인가", `git diff --staged`는 "커밋할 내용이 무엇인가"를 확인하는 명령어입니다.
