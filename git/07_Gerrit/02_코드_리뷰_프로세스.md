# Gerrit 코드 리뷰 프로세스

코드 리뷰는 단순히 코드를 확인하는 것을 넘어, 변경의 품질과 안정성을 보장하는 체계적인 프로세스입니다. 많은 개발자들이 코드 리뷰를 부담스러워하지만, 실제로는 팀 전체의 코드 품질을 높이고 지식을 공유하는 중요한 협업 도구입니다. 우리는 이 장에서 Gerrit의 코드 리뷰 프로세스를 단계별로 살펴보고, 리뷰어와 개발자 간의 효과적인 협업 방법을 알아보겠습니다. 코드 리뷰를 체계적으로 이해하면 팀의 코드 품질을 높이고 예상치 못한 버그를 사전에 방지할 수 있습니다.


## 👨‍💻 실전 프로젝트: 코드 리뷰 워크플로우 체험

이번 실전 프로젝트에서는 Gerrit의 코드 리뷰 워크플로우를 처음부터 끝까지 직접 체험해보겠습니다. 여러분은 먼저 간단한 기능을 구현하여 Gerrit에 Change를 생성하고, 이후 다른 팀원(또는 자신)이 리뷰어가 되어 코드 리뷰를 수행합니다. 리뷰 코멘트를 작성하고, 피드백을 반영한 후 최종적으로 승인 및 병합까지 완료하는 전 과정을 경험할 수 있습니다. 이 프로젝트를 완료하면 Gerrit을 사용한 협업 개발의 전체 흐름을 완벽히 이해하게 될 것입니다.

실습을 위한 사전 준비로 Gerrit 저장소가 준비되어 있어야 합니다. 아래 명령어를 따라 직접 실습을 진행해보시기 바랍니다.

```bash
# 1. 개발자: 기능 구현 및 push
$ git clone ssh://username@gerrit.example.com:29418/my-project
$ cd my-project
$ scp -p -P 29418 username@gerrit.example.com:hooks/commit-msg .git/hooks/
$ chmod +x .git/hooks/commit-msg
$ git switch -c feature/user-profile

# 사용자 프로필 페이지 구현
$ cat > profile.html << 'EOF'
<h1>사용자 프로필</h1>
<div id="profile">
  <p>이름: <span id="name"></span></p>
  <p>이메일: <span id="email"></span></p>
</div>
EOF

$ git add profile.html
$ git commit -m "사용자 프로필 페이지 추가

사용자 정보를 표시하는 기본 프로필 페이지를 구현하였습니다.

Change-Id: Iabcdef1234567890abcdef1234567890abcdef12"

# 리뷰 요청 (리뷰어 alice 지정)
$ git push origin HEAD:refs/for/main%r=alice@example.com

# 2. 리뷰어: Gerrit 웹 UI에서 리뷰 수행
# https://gerrit.example.com/c/my-project/+/124 접속
# 코드 확인 후 Code-Review +1 (의견 있음) 부여
# "프로필에 아바타 이미지 영역이 있으면 좋겠습니다." 코멘트 작성

# 3. 개발자: 피드백 반영
$ echo "  <p>아바타: <img id='avatar' src='' /></p>" >> profile.html
$ git add profile.html
$ git commit --amend --no-edit
$ git push origin HEAD:refs/for/main

# 4. 리뷰어: 최종 승인
# Code-Review +2, Verified +1 → Submit 버튼 활성화
# Submit 클릭 → main 브랜치에 병합 완료!
```

이 실습을 통해 Gerrit의 코드 리뷰가 단순한 버그 검출을 넘어 팀 내 지식 공유와 코드 일관성 유지에 얼마나 도움이 되는지 체감할 수 있을 것입니다.


## 학습 목표

- Gerrit 코드 리뷰의 전체 라이프사이클을 이해하고 설명할 수 있다
- 리뷰어의 역할과 책임, 효과적인 리뷰 코멘트 작성법을 익힌다
- Code-Review와 Verified 라벨의 부여 기준을 설명할 수 있다
- 리뷰 거절 상황에 대응하는 방법을 이해한다

## 리뷰 프로세스 개요

## 리뷰어의 역할

리뷰 프로세스의 개요를 이해했다면, 이제 리뷰어가 실제로 어떤 책임을 가지고 리뷰를 수행해야 하는지 알아보겠습니다. 리뷰어는 단순히 코드의 오류를 찾는 것을 넘어, 팀의 코드 품질과 일관성을 유지하는 중요한 역할을 담당합니다. 체계적인 체크리스트를 활용하면 놓치는 부분 없이 꼼꼼하게 리뷰할 수 있습니다.

### 리뷰어 체크리스트

```
□ 코드 스타일이 팀 컨벤션을 따르는가?
□ 버그나 잠재적 문제가 없는가?
□ 테스트 코드가 포함되었는가?
□ 문서가 업데이트되었는가?
□ 보안 취약점은 없는가?
□ 성능에 영향을 주는가?
□ 변경이 너무 크지 않은가? (적절한 단위)
```

### 리뷰 코멘트 작성 예시

좋은 리뷰 코멘트는 구체적이고 건설적이어야 합니다. 아래 예시를 통해 효과적인 피드백과 그렇지 않은 피드백을 비교해보겠습니다. 단순히 "수정하세요"라고 말하는 것은 개발자에게 어떤 부분을 어떻게 개선해야 할지 전혀 알려주지 못합니다.

```
# ❌ 도움이 되지 않는 코멘트
"이 코드는 별로예요"
"수정하세요"

# ✅ 도움이 되는 코멘트
"이 부분은 변수명이 `pric`으로 되어 있는데 `price`가 맞는 것 같습니다.
  39번 줄에서 `price`를 사용하고 있으니 통일하는 것이 좋겠습니다."

"28-35번 줄의 중복 로직이 보입니다. `calculateTotal()` 함수로
  추출하면 가독성이 좋아질 것 같습니다."

"이 함수는 50줄이 넘습니다. 작은 함수들로 분리하는 것이
  유지보수에 좋을 것 같습니다."
```

효과적인 코멘트를 작성할 때는 "무엇이 문제인지", "왜 문제인지", "어떻게 개선할 수 있는지"의 세 가지 요소를 포함하는 것이 좋습니다. 이렇게 하면 개발자가 리뷰어의 의도를 정확히 이해하고 더 나은 방향으로 코드를 개선할 수 있습니다.

## 리뷰 라벨 부여 기준

지금까지 리뷰어의 역할과 좋은 코멘트 작성법에 대해 배웠습니다. 다음으로 각 리뷰 라벨을 부여하는 구체적인 기준을 알아보겠습니다. 라벨을 올바르게 부여하려면 명확한 기준이 필요합니다.

### Code-Review +2 승인 기준

```markdown
# +2를 주려면 다음 조건을 만족해야 함
1. 코드가 정확하고 버그가 없음
2. 코딩 컨벤션 준수
3. 적절한 테스트 포함
4. 문서 변경이 필요한 경우 함께 포함
5. 변경 범위가 적절함

# +1 조건
코드가 대체로 괜찮지만, 사소한 의견이 있음

# -1 조건
수정이 필요하지만, 전체적인 방향은 괜찮음

# -2 조건 (강력 반대)
변경 자체의 방향이 잘못되었거나, 심각한 문제가 있음
```

### Verified (CI) 기준

Verified 라벨은 CI/CD 파이프라인의 테스트 결과를 반영합니다. 사람의 판단이 아닌 자동화된 테스트 결과를 기반으로 부여되므로 객관적인 품질 지표 역할을 합니다. 아래는 Jenkins를 사용한 CI 파이프라인 예시입니다.

```yaml
# +1 Verified (Jenkinsfile 예시)
pipeline {
    agent any
    stages {
        stage('Build') {
            steps { sh 'npm run build' }
        }
        stage('Test') {
            steps {
                sh 'npm test'
                junit 'test-results/**/*.xml'
            }
        }
        stage('Lint') {
            steps { sh 'npm run lint' }
        }
    }
}
```

## 리뷰 댓글 명령어

리뷰 라벨의 기준을 이해했다면, 이제 실제 리뷰를 진행할 때 사용할 수 있는 Gerrit의 댓글 명령어를 알아보겠습니다. Gerrit 웹 UI에서 댓글에 특수 명령어를 사용할 수 있습니다. 이를 통해 효율적으로 리뷰를 진행할 수 있습니다.

```
# 리뷰어 명령어
/patch set level comment       # 전체 Patch Set에 코멘트
/inline comment                # 특정 코드 라인에 코멘트
/reply                         # 답글

# 점수 부여 (드롭다운 메뉴)
Verified: +1 / -1
Code-Review: +2 / +1 / -1 / -2

# 작업 명령
/abandon                       # Change 포기 (작성자가 사용)
/restore                       # 포기된 Change 복원
/submit                        # 병합 실행 (권한 필요)
/rebase                        # 최신 target 브랜치로 리베이스
```

## 여러 리뷰어 협업

리뷰 댓글과 명령어를 활용하는 방법을 배웠습니다. 이제 더 복잡한 상황, 즉 여러 명의 리뷰어가 함께 협업하는 시나리오를 살펴보겠습니다. 대규모 변경이나 중요한 변경은 여러 리뷰어가 각자의 전문 분야를 나누어 검토하는 것이 효과적입니다.

```bash
# 대규모 변경: 여러 리뷰어 지정
$ git push origin HEAD:refs/for/main%r=alice@example.com,r=bob@example.com,cc=charlie@example.com

# 리뷰어별 역할
alice@example.com   → 보안 전문가 (보안 리뷰)
bob@example.com     → 시니어 개발자 (일반 리뷰)
charlie@example.com → CC (참고만)
```

여러 리뷰어가 참여할 때는 각 리뷰어의 전문성을 고려하여 역할을 분담하는 것이 중요합니다. 예를 들어 보안에 민감한 변경이라면 보안 전문가를 리뷰어로 지정하고, 아키텍처 전반에 영향을 주는 변경이라면 시니어 개발자가 참여하도록 합니다.

## 리뷰 거절 대응

여러 리뷰어 협업 상황에서도 리뷰가 거절되는 경우는 발생할 수 있습니다. 리뷰어가 -2를 주어 변경이 거절된 경우, 당황하지 말고 체계적으로 대응하는 것이 중요합니다. -2는 단순히 "싫다"는 의미가 아니라 "현재 상태로는 병합될 수 없다"는 엄중한 신호입니다.

```bash
# 리뷰어가 -2를 주고 변경을 거절한 경우

# 1. 코멘트 확인
# "이 접근 방식은 확장성이 부족합니다. 디자인 패턴을 고려해보세요."

# 2. 로컬에서 수정
$ git switch feature/login
# 완전히 새로운 접근 방식으로 코드 재작성

# 3. 동일 Change에 새로운 Patch Set push
$ git add .
$ git commit --amend -m "로그인 페이지 추가 (리팩토링)

디자인 패턴 적용으로 확장성 개선

Change-Id: I123..."  # 동일 Change-ID 유지
$ git push origin HEAD:refs/for/main

# 또는 Change를 포기하고 새로 시작
$ git push origin HEAD:refs/for/main%abandon
$ git reset HEAD~3  # 이전 커밋들 제거
# 새 브랜치에서 새로 시작
```

리뷰 거절을 경험했을 때 가장 중요한 것은 방어적으로 반응하지 않고 리뷰어의 피드백을 성장의 기회로 받아들이는 태도입니다. 건설적인 비판은 팀의 코드 품질을 높이는 데 필수적인 요소입니다.

## Gerrit 대시보드 활용

리뷰 거절 상황에 대응하는 방법을 배웠습니다. 이제 Gerrit의 대시보드를 효과적으로 활용하여 리뷰 상태를 관리하는 방법을 알아보겠습니다. Gerrit 대시보드는 현재 진행 중인 모든 리뷰의 상태를 한눈에 보여줍니다. 대시보드를 효과적으로 활용하면 리뷰가 지연되는 상황을 방지할 수 있습니다.

```
My Reviews (내 리뷰 대상):
  ◇ Change 128: 설정 파일 수정              [PS2]
    Owner: you    |  Status: Needs Review
    +1 Code-Review (alice)
    -1 Verified (Jenkins)     ← CI 실패!

My Outgoing Reviews (내가 보낸 리뷰):
  ◆ Change 129: 검색 기능 추가              [PS3]
    Owner: you    |  Status: Awaiting Reviewer
    +2 Code-Review (bob)
    +1 Verified (Jenkins)
    → Ready to Submit! 🎉

Recently Closed:
  ✓ Change 120: README 업데이트              Merged
  ✗ Change 121: 실험적 기능                   Abandoned
```

## 리뷰 속도 높이는 팁

Gerrit 대시보드를 통해 리뷰 상태를 모니터링하는 방법을 배웠습니다. 마지막으로 리뷰 속도를 높이기 위한 실용적인 팁을 알아보겠습니다. 리뷰 속도는 팀의 전체 개발 속도에 직접적인 영향을 미칩니다. 변경 작성자와 리뷰어 각자가 실천할 수 있는 팁을 정리하였습니다.

### 변경 작성자:
```bash
# 1. 변경을 작게 유지 (1시간 내 리뷰 가능한 크기)
# 2. 커밋 메시지에 변경 이유를 명확히 작성
# 3. 리뷰어를 미리 지정
$ git push origin HEAD:refs/for/main%r=alice@example.com,topic=login

# 4. CI 통과 후 리뷰 요청
# 5. 리뷰 코멘트에 빠르게 응답
```

### 리뷰어:
```bash
# 1. 하루에 2회 이상 리뷰 대시보드 확인
# 2. 24시간 이내에 첫 리뷰 완료 목표
# 3. 명확하고 건설적인 피드백
# 4. 작은 변경은 우선 처리
```

## 한눈에 정리

지금까지 배운 Gerrit 코드 리뷰 프로세스의 핵심 내용을 한눈에 정리하면 다음과 같습니다. 이 표를 참고하면 프로세스의 각 단계와 라벨의 의미를 빠르게 복습할 수 있습니다.

| 개념 | 설명 |
|------|------|
| 리뷰 라이프사이클 | Push → 리뷰 → 수정 → 승인 → Submit |
| Code-Review +2 | 병합 가능, 모든 체크리스트 충족 필요 |
| Code-Review -2 | 강력 반대, 병합 불가능 |
| Verified +1 | CI/테스트 통과 |
| Verified -1 | CI/테스트 실패 |
| 효과적인 코멘트 | 구체적이고 건설적인 피드백 제공 |
| 리뷰 거절 대응 | 코멘트 확인 → 수정 → 새 Patch Set Push |

## 연습 문제

지금까지 배운 내용을 바탕으로 다음 연습 문제를 풀어보면서 자신의 이해도를 점검해보시기 바랍니다. 각 문제는 실제 Gerrit 사용 시 마주칠 수 있는 상황을 반영하였습니다.

1. Gerrit 코드 리뷰 프로세스에서 개발자가 push한 후 최종적으로 변경 사항이 main 브랜치에 병합되기까지의 단계를 순서대로 설명하시오.

2. Code-Review +2와 -2는 각각 어떤 상황에서 부여하는지 구체적인 기준을 서술하시오.

3. 리뷰어가 "이 접근 방식은 확장성이 부족합니다. 디자인 패턴을 고려해보세요."라고 코멘트를 남기고 -2를 부여했습니다. 개발자가 취해야 할 조치를 git 명령어와 함께 설명하시오.

📌 정답 및 해설

**문제 1 정답 및 해설:**

Gerrit 코드 리뷰 프로세스에서 개발자가 push한 후 최종적으로 변경 사항이 main 브랜치에 병합되기까지의 단계는 다음과 같습니다. 1단계: 개발자가 `git push origin HEAD:refs/for/main`으로 변경 사항을 Gerrit에 푸시합니다. 이 단계에서 Commit-msg hook이 생성한 Change-Id가 커밋 메시지에 포함되어 있어야 합니다. 2단계: Gerrit이 새로운 Change(리뷰 요청)를 생성하고, 리뷰어에게 리뷰 요청이 전달됩니다. 3단계: 리뷰어가 코드를 검토하고 Code-Review 점수(-2, -1, +1, +2)를 부여합니다. 4단계: CI 시스템이 자동으로 빌드와 테스트를 실행하고 Verified 점수(+1, -1)를 부여합니다. 5단계: Code-Review +2와 Verified +1을 모두 획득하면 Change가 "Submit 가능" 상태가 됩니다. 6단계: 리뷰가 완료된 Change를 개발자 또는 리뷰어가 "Submit" 버튼을 클릭하여 main 브랜치에 병합합니다. 이 단계에서 Gerrit은 자동으로 변경 사항을 main 브랜치에 적용하고 Change를 "Merged" 상태로 변경합니다.

**문제 2 정답 및 해설:**

Code-Review +2는 "이 변경 사항은 승인되었으며, 아무 문제없이 병합할 수 있습니다"라는 의미입니다. +2는 일반적으로 프로젝트의 메인테이너나 시니어 리뷰어만 부여할 수 있으며, 한 번의 +2 승인으로 병합이 가능합니다. +2를 부여하기 위한 기준은 코드가 정확하고 버그가 없으며, 코딩 컨벤션을 준수하고, 적절한 테스트가 포함되어 있으며, 보안 문제가 없고, 전반적인 아키텍처 관점에서도 문제가 없다고 판단될 때입니다. 반면 Code-Review -2는 "이 변경 사항은 현재 상태로 병합되어서는 안 되며, 반드시 수정이 필요합니다"라는 의미입니다. -2는 +2와 마찬가지로 권한 있는 리뷰어만 부여할 수 있으며, 한 번의 -2로 병합이 차단됩니다. -2를 부여하는 기준은 심각한 버그, 보안 취약점, 잘못된 아키텍처 결정, 또는 프로젝트의 핵심 방향성에 위배되는 변경이 있을 때입니다. -2는 말 그대로 "거부"에 해당하므로, 부여하기 전에 신중한 판단이 필요합니다.

**문제 3 정답 및 해설:**

리뷰어가 -2를 부여하고 확장성 문제를 지적했을 때, 개발자는 다음과 같이 대응해야 합니다. 1단계: 리뷰어의 코멘트를 면밀히 분석하고, 현재 접근 방식의 한계를 이해합니다. 필요한 경우 리뷰어에게 질문하여 구체적인 의견을 추가로 요청합니다. 2단계: 리뷰어의 제안을 바탕으로 더 확장성 있는 디자인 패턴(예: Strategy 패턴, Factory 패턴 등)을 적용하여 코드를 재설계합니다. 3단계: 로컬에서 코드를 수정한 후 `git commit --amend`로 기존 커밋을 수정합니다(--amend를 사용하면 동일한 Change-Id가 유지됨). 4단계: `git push origin HEAD:refs/for/main`으로 Gerrit에 새로운 Patch Set을 푸시합니다. 5단계: Gerrit 웹 UI에서 리뷰어에게 "코멘트 감사합니다. Strategy 패턴을 적용하여 확장성을 개선했습니다. 다시 리뷰 부탁드립니다"라는 답글을 남깁니다. 6단계: 리뷰어가 다시 리뷰를 수행하고, 문제가 해결되었다고 판단되면 Code-Review +2를 부여합니다. 여기서 중요한 점은 단순히 코드만 수정하는 것이 아니라, 리뷰어의 코멘트에 대한 답변을 통해 소통하면서 개선 방향을 투명하게 공유하는 것입니다.
