# 유용한 Git 도구

Git은 명령줄 도구 자체로도 강력하지만, 함께 사용하면 더욱 생산성을 높여주는 다양한 도구들이 있습니다. 이 장에서는 Git 사용을 더 편리하고 효율적으로 만들어주는 유용한 도구들을 소개합니다.

Git 기본 명령어만으로도 대부분의 작업을 수행할 수 있지만, 전문 도구를 사용하면 작업 속도와 편의성이 비약적으로 향상됩니다. 마치 나무를 자를 때 맨손으로 하는 것과 전기톱을 사용하는 것의 차이와 같습니다. Git 기초 명령어를 이미 익혔다면, 이 장에서 소개하는 도구들을 통해 여러분의 Git 활용 능력을 한 단계 더 끌어올릴 수 있습니다.

👨‍💻 **실전 프로젝트: Git 도구 체험하기**

이론만으로는 각 도구의 진가를 체감하기 어렵습니다. 가장 좋은 방법은 직접 설치하고 사용해 보는 것입니다. 지금부터 tig와 lazygit이라는 두 가지 대표적인 Git 도구를 설치하고 기본적인 사용법을 체험해 보겠습니다.

```bash
# === 1. tig 설치 및 기본 사용 ===

# 운영 체제에 맞게 tig를 설치합니다
# macOS
$ brew install tig

# Ubuntu/Debian
$ sudo apt install tig

# 설치가 완료되었으면, Git 저장소가 있는 폴더로 이동합니다
$ cd my-project

# tig를 실행하면 커밋 히스토리가 대화형 화면으로 표시됩니다
$ tig

# tig 화면에서 할 수 있는 기본 동작:
# - j/k 키로 위아래 이동
# - Enter 키로 선택한 커밋의 상세 diff 확인
# - / 키로 커밋 메시지 검색
# - q 키로 종료

# === 2. lazygit 설치 및 기본 사용 ===

# macOS
$ brew install lazygit

# Ubuntu/Debian
$ sudo add-apt-repository ppa:lazygit-team/release
$ sudo apt update
$ sudo apt install lazygit

# Git 저장소 폴더에서 lazygit 실행
$ cd my-project
$ lazygit

# lazygit 화면에서 할 수 있는 기본 동작:
# - 화살표 키로 파일/커밋 선택
# - Space 키로 Staging/Unstaging
# - c 키로 커밋 메시지 작성
# - p 키로 Push
```

이렇게 tig와 lazygit을 설치하고 기본 동작을 확인해 보았습니다. 두 도구 모두 터미널에서 동작하지만, tig는 히스토리 탐색에 강점이 있고 lazygit은 Staging, 커밋, 브랜치 관리 등 전반적인 Git 작업을 GUI처럼 처리하는 데 강점이 있습니다. 이제 각 도구를 더 자세히 살펴보겠습니다.

## 학습 목표

- Git 관련 유용한 도구의 종류와 특징을 이해한다
- `tig`를 활용하여 터미널에서 효율적으로 Git 히스토리를 탐색할 수 있다
- 상황에 맞는 적절한 Git 도구를 선택하여 사용할 수 있다

## 1. tig: 터미널용 Git 브라우저

`tig`는 터미널에서 동작하는 텍스트 모드 Git 브라우저입니다. 커밋 로그, 브랜치, diff 등을 키보드만으로 빠르게 탐색할 수 있습니다. 마치 `git log --oneline --graph --all`을 대화형으로 사용하는 것과 같습니다.

tig의 가장 큰 장점은 "손이 키보드에서 떠날 필요가 없다"는 점입니다. 마우스를 전혀 사용하지 않고 키보드 단축키만으로 모든 Git 히스토리를 탐색할 수 있습니다. 이는 개발 중에 "이 커밋이 언제 만들어졌지?", "이 파일은 누가 마지막으로 수정했지?"와 같은 질문에 즉시 답을 얻을 수 있게 해 줍니다. 터미널 환경에서 작업하는 개발자에게는 마치 브라우저의 뒤로 가기 버튼과 같은 존재입니다.

### 설치

```bash
# macOS
$ brew install tig

# Ubuntu/Debian
$ sudo apt install tig

# Windows (Git Bash)
$ choco install tig
```

### 실행 화면 예시

```text
$ tig

  Browse branch 'main'                           +14:30
  [main] a1b2c3d Add login page                  홍길동 2 min ago
  [main] d4e5f6f Fix payment bug                 홍길동 1 hour ago
  [main] g7h8i9j Add payment module              홍길동 3 hours ago
  [main] k1l2m3n Initial commit                  홍길동 1 day ago
  [main] a1b2c3d Add login page
  ─────────────────────────────────────────────────────────
  diff --git a/login.html b/login.html
  new file mode 100644
  @@ -0,0 +1,10 @@
  +<h1>Login</h1>
  +<form>
  +  <input type="text" name="username">
  +  <input type="password" name="password">
  +  <button>Submit</button>
  +</form>

  [main] a1b2c3d - Add login page               +14:30
  [status] [logs] [diff] [tree] [blame] [help]
```

위 화면에서 `↓`/`↑` 키로 커밋을 선택하고 `Enter`를 누르면 diff를 볼 수 있습니다. `l` 키를 누르면 전체 로그 뷰로, `t` 키를 누르면 트리 뷰(파일 목록)로 전환됩니다.

tig의 화면은 크게 두 부분으로 나뉩니다. 상단에는 커밋 목록과 브랜치 정보가 표시되고, 하단에는 선택한 커밋의 상세 내용이 표시됩니다. 상하단의 분할 비율은 화면 크기에 따라 자동으로 조정되며, `Tab` 키를 누르면 상하단 간 포커스를 전환할 수 있습니다.

### 주요 사용법

```bash
# 커밋 로그 브라우저 열기 (가장 많이 사용)
$ tig

# 특정 파일의 히스토리 보기
$ tig -- README.md

# 브랜치 간 비교
$ tig main..feature/login

# Stash 목록 보기
$ tig stash

# grep 결과를 tig로 보기
$ tig grep "function"
```

`tig main..feature/login` 명령어는 `main` 브랜치에는 없고 `feature/login` 브랜치에만 있는 커밋들을 보여줍니다. 이는 브랜치 간 차이를 분석할 때 매우 유용합니다. 또한 `tig grep` 명령어는 프로젝트 전체에서 특정 문자열이 포함된 커밋을 검색할 때 사용합니다.

### tig 단축키

`tig`를 실행한 후 사용할 수 있는 주요 키보드 단축키입니다:

| 키 | 기능 |
|---|---|
| `j` / `k` | 위/아래로 이동 |
| `Enter` | 선택한 커밋의 상세 내용(diff) 보기 |
| `l` | 로그 뷰로 전환 |
| `d` | diff 뷰로 전환 |
| `t` | 트리 뷰로 전환 (파일 목록) |
| `/` | 검색 |
| `m` | 현재 브랜치에서 merge |
| `M` | rebase |
| `r` | refresh (새로고침) |
| `q` | 뒤로 가기 / 종료 |
| `h` | 도움말 |

이 단축키들은 Vim 에디터의 키 바인딩에 익숙한 사용자라면 더욱 쉽게 적응할 수 있습니다. `j`/`k`로 이동하고, `/`로 검색하는 방식이 Vim과 동일하기 때문입니다. 각 뷰 간 전환(`l`, `d`, `t`)을 마스터하면 tig를 자유자재로 다룰 수 있습니다.

### 실전 예시

```bash
$ cd my-project
$ tig
# ↓ j 키로 로그 탐색
# ↓ Enter 키로 diff 확인
# ↓ / 키로 "버그" 검색
# ↓ n 키로 다음 검색 결과로 이동
```

이 예시는 실제 코드 리뷰 상황에서 매우 유용합니다. "버그"와 관련된 커밋을 찾고 싶다면, `tig`를 실행한 후 `/` 키를 누르고 "버그"를 입력한 후 `Enter`를 누르면 됩니다. `n` 키를 누르면 다음 검색 결과로, `N` 키를 누르면 이전 검색 결과로 이동합니다. 이 모든 과정이 키보드만으로 이루어지므로 작업 효율이 극대화됩니다.

## 2. lazygit: 터미널용 Git GUI

tig가 히스토리 탐색에 특화되어 있다면, lazygit은 Git의 거의 모든 작업을 터미널에서 GUI처럼 처리할 수 있는 종합 도구입니다.

`lazygit`는 터미널에서 동작하는 더 직관적인 Git GUI 도구입니다. 화살표 키로 Staging, 커밋, 브랜치 전환, 병합 등을 모두 처리할 수 있습니다.

lazygit의 가장 큰 장점은 Git 명령어를 전혀 외우지 않아도 된다는 점입니다. Staging, 커밋, 브랜치 생성 및 전환, 병합, Push, Pull, 충돌 해결까지 모든 작업을 화살표 키와 몇 개의 단축키로 수행할 수 있습니다. 초보자에게는 Git을 시각적으로 이해하는 데 큰 도움이 되고, 숙련자에게는 반복적인 타이핑에서 해방시켜 줍니다.

```bash
# 설치
# macOS
$ brew install lazygit

# Ubuntu/Debian
$ sudo add-apt-repository ppa:lazygit-team/release
$ sudo apt update
$ sudo apt install lazygit

# 실행
$ lazygit
```

### 실행 화면 예시

```text
  ┌─ Status ─────────────────────────────┐ ┌─ Staging Area ────────────────────┐
  │  ↑↓ my-project                       │ │                                   │
  │                                      │ │   ■ M index.html                  │
  │  ? ─ Recent repositories             │ │   ■ M style.css                   │
  │                                      │ │                                   │
  │  ┌─ Commits ─────────────────────┐   │ │                                   │
  │  │ a1b2c3d Add login page        │   │ │                                   │
  │  │ d4e5f6f Fix payment bug       │   │ │                                   │
  │  │ g7h8i9j Add payment module    │   │ │                                   │
  │  │ k1l2m3n Initial commit        │   │ └───────────────────────────────────┘
  │  └───────────────────────────────┘   │ ┌─ Main ────────────────────────────┐
  │                                      │ │  1: unstaged changes              │
  │  ┌─ Branches ───────────────────┐    │ │  ?? index.html                    │
  │  │ * main                       │    │ │  ?? style.css                     │
  │  │   feature/login              │    │ └───────────────────────────────────┘
  │  └──────────────────────────────┘    └──────────────────────────────────────
```

화살표 키로 파일을 선택하고 `Space`로 Staging/Unstaging, `c`로 커밋 메시지 작성, `p`로 push까지 한 화면에서 처리할 수 있습니다.

lazygit의 화면은 여러 개의 패널로 구성되어 있습니다. 왼쪽 상단부터 시계 방향으로 Status(상태), Files(파일 목록), Branches(브랜치 목록), Commits(커밋 목록), Staging Area(스테이징 영역) 순서로 배치되어 있습니다. 각 패널 간 이동은 숫자 키(1~5)로 할 수 있으며, 현재 선택된 패널은 테두리가 강조 표시됩니다.  특히 충돌(Conflict)이 발생했을 때 lazygit은 충돌이 발생한 파일을 시각적으로 표시하고, 내장된 diff 도구로 충돌을 해결할 수 있는 기능을 제공합니다.

## 3. gh: GitHub CLI

터미널에서의 Git 작업을 넘어, 이제 GitHub의 기능을 명령줄에서 직접 제어하고 싶다면 `gh`가 정답입니다.

`gh`는 GitHub을 명령줄에서 제어할 수 있는 공식 CLI 도구입니다.

GitHub의 웹 인터페이스는 직관적이지만, 반복적인 작업(예: PR 목록 확인, 이슈 검색, PR 생성 및 병합)을 할 때마다 브라우저로 전환하는 것은 생산성을 크게 떨어뜨립니다. `gh`를 사용하면 브라우저로 전환할 필요 없이 터미널에서 모든 GitHub 작업을 처리할 수 있습니다. 특히 코드 리뷰나 CI/CD 모니터링과 같이 Git과 GitHub를 연속적으로 사용하는 작업 흐름에서 `gh`는 빛을 발합니다.

```bash
# 설치
# macOS
$ brew install gh

# Ubuntu/Debian (GitHub CLI 공식 저장소)
$ sudo apt install gh

# 로그인
$ gh auth login
```

### 실행 화면 예시

```text
$ gh pr list --limit 3
Showing 3 of 3 open pull requests in jingu279/Document

#51  Add login form              feature/login     about 2 hours ago
#49  Fix payment calculation bug  hotfix/payment     about 1 day ago
#48  Update README                docs/readme        about 3 days ago

$ gh pr view 51

Assignee: jingu279 • 1 reviewer requested

Adds login form with email and password fields.
Includes validation and error handling.

+120 -0 in 3 files

$ gh issue list

#11  Add search functionality         enhancement  about 2 days ago
#10  Update dependencies              chore        about 3 days ago
```

`gh auth login` 명령어를 실행하면 브라우저가 열리면서 GitHub 인증을 진행합니다. 한 번 인증이 완료되면 이후에는 모든 `gh` 명령어를 자유롭게 사용할 수 있습니다. `gh`는 GitHub Actions의 워크플로우 실행 상태를 실시간으로 모니터링하는 `gh run watch`와 같은 고급 기능도 제공합니다.

### 주요 명령어

```bash
$ gh repo create my-project    # GitHub에 저장소 생성
$ gh pr create -f              # PR 생성 (자동 제목)
$ gh pr review --approve       # PR 승인
$ gh pr merge                  # PR 병합
$ gh run watch                 # Actions 실행 실시간 확인
$ gh issue list --label bug    # 특정 라벨의 이슈 검색
```

`gh pr create -f` 명령어는 현재 브랜치의 커밋 메시지를 기반으로 자동으로 PR 제목과 본문을 생성합니다. 이는 브랜치 이름과 커밋 메시지를 일관성 있게 관리하는 팀에서 특히 유용합니다. 또한 `gh run watch`는 CI/CD 파이프라인의 실행 상태를 터미널에서 실시간으로 확인할 수 있게 해 줍니다.

## 4. diff-so-fancy: 예쁜 diff

`git diff` 명령어의 출력은 기능적으로는 완벽하지만, 가독성 측면에서는 개선의 여지가 많습니다. `diff-so-fancy`는 이 문제를 해결합니다.

`diff-so-fancy`는 `git diff`의 출력을 더 읽기 쉽게 보여주는 도구입니다.

코드 리뷰에서 diff를 읽는 것은 핵심적인 작업입니다. 변경된 파일이 많거나 diff의 길이가 길어지면 기본 `git diff`의 출력은 눈에 잘 들어오지 않습니다. `diff-so-fancy`는 파일명을 강조 표시하고, 변경된 청크(Chunk)의 헤더를 간결하게 표현하며, 추가/삭제된 라인을 더 선명하게 구분해 줍니다. 이러한 가독성 향상은 코드 리뷰의 속도와 정확성을 모두 높여 줍니다.

```bash
# 설치
$ brew install diff-so-fancy   # macOS
$ npm install -g diff-so-fancy  # npm

# Git 설정에 적용
$ git config --global core.pager "diff-so-fancy | less --tabs=4 -RFX"
$ git config --global interactive.diffFilter "diff-so-fancy --patch"
```

### 적용 전후 비교

```text
# 적용 전 (기본 git diff):
─────────────────────────────────────────────────────────
diff --git a/index.html b/index.html
index a1b2c3d..d4e5f6g 100644
--- a/index.html
+++ b/index.html
@@ -10,7 +10,9 @@
     <title>내 웹사이트</title>
   </head>
   <body>
+    <h1>환영합니다!</h1>
     <p>내용입니다.</p>

# 적용 후 (diff-so-fancy):
─────────────────────────────────────────────────────────
index.html ──────────────────────────────────────────────
@¡@ 10,7 10,9 @¡@
     <title>내 웹사이트</title>
   </head>
   <body>
+    <h1>환영합니다!</h1>
     <p>내용입니다.</p>
```

변경된 파일명이 강조 표시되고, 청크 헤더가 간결해져서 가독성이 크게 향상됩니다.

`diff-so-fancy`의 가장 눈에 띄는 개선점은 파일명이 강조되어 표시된다는 점입니다. 기본 `git diff`에서는 파일명이 일반 텍스트와 거의 구분되지 않지만, `diff-so-fancy`에서는 파일명이 선명하게 강조됩니다. 또한 변경된 라인의 `+`와 `-` 기호가 색상 코드와 함께 더 선명하게 표시되어, 어떤 라인이 추가되고 삭제되었는지를 한눈에 파악할 수 있습니다.

## 5. gitk: GUI 히스토리 뷰어

앞서 소개한 도구들은 모두 별도로 설치해야 하지만, 이번에 소개할 도구는 Git에 기본 내장되어 있어 별도의 설치 과정이 필요 없습니다.

`gitk`는 Git에 내장된 그래픽 히스토리 뷰어입니다. 별도 설치가 필요 없습니다.

`gitk`의 가장 큰 장점은 "별도 설치가 필요 없다"는 점입니다. Git만 설치되어 있다면 어디서든 실행할 수 있습니다. 그래픽 인터페이스이므로 마우스로 클릭하면서 히스토리를 탐색할 수 있어, GUI 환경에 익숙한 사용자나 Git 초보자에게 적합합니다.

```bash
$ gitk
```

### 실행 화면 예시

```text
  ┌─────────────────────────────────────────────────────────────┐
  │  [●] a1b2c3d  Add login page                 main,* HEAD   │
  │  │                                                     │   │
  │  [●] d4e5f6f  Fix payment bug                             │   │
  │  │                                                     │   │
  │  [●] g7h8i9j  Add payment module            feature/pay   │   │
  │  │/                                                      │   │
  │  [●] k1l2m3n  Initial commit                             │   │
  │                                                          │   │
  │  ────────────────────────────────────────────────────    │   │
  │  a1b2c3d Add login page                                  │   │
  │                                                          │   │
  │  diff --git a/login.html b/login.html                    │   │
  │  index 0000000..e69de29                                  │   │
  │  --- /dev/null                                           │   │
  │  +++ b/login.html                                        │   │
  └─────────────────────────────────────────────────────────────┘
```

상단 패널에서 히스토리 그래프를 시각적으로 확인하고, 하단 패널에서 선택한 커밋의 diff를 볼 수 있습니다. 마우스 클릭으로 탐색할 수 있어 초보자에게 직관적입니다.

`gitk`는 특히 브랜치 구조가 복잡한 프로젝트에서 빛을 발합니다. 여러 브랜치가 분기되고 병합되는 과정을 그래프로 시각화하여 보여주기 때문에, 프로젝트의 전체적인 개발 흐름을 한눈에 파악할 수 있습니다. 또한 검색 기능을 통해 특정 문자열이 포함된 커밋을 찾거나, 특정 파일의 히스토리만 필터링하여 볼 수도 있습니다.

## 6. git-gui: GUI 커밋 도구

`gitk`가 히스토리 확인에 특화되어 있다면, `git-gui`는 커밋을 만드는 과정에 특화된 도구입니다.

`git-gui`는 Git에 내장된 GUI 도구로, Staging과 커밋을 시각적으로 수행할 수 있습니다.

`git-gui`는 Staging Area의 개념을 시각적으로 이해하는 데 큰 도움이 됩니다. 어떤 파일이 Staging되었고, 어떤 파일이 아직 Staging되지 않았는지가 두 개의 패널로 명확히 구분되어 표시됩니다. 또한 diff 패널에서 변경된 내용을 실시간으로 확인하면서 커밋할 파일을 선택할 수 있어, 실수로 원하지 않는 파일을 커밋하는 것을 방지할 수 있습니다.

```bash
$ git gui
```

```text
  ┌─ Unstaged Changes ────────┐ ┌─ Staged Changes ──────────┐
  │  index.html               │ │  style.css                │
  │  app.js                   │ │                           │
  │                           │ │                           │
  │  [>>] Stage                │ │  [<<] Unstage             │
  └───────────────────────────┘ └───────────────────────────┘
  ┌─ Commit Message ────────────────────────────────────────┐
  │  [로그인 페이지 추가]                                      │
  │                                                          │
  │  [Commit]  [ Amend ]  [ Sign Off ]                       │
  └──────────────────────────────────────────────────────────┘
  ┌─ Diff (index.html) ─────────────────────────────────────┐
  │  +<h1>환영합니다!</h1>                                   │
  └──────────────────────────────────────────────────────────┘
```

왼쪽에서 unstaged 파일을 선택하고 Stage 버튼을 누른 후, 커밋 메시지를 입력하고 Commit 버튼을 누르면 됩니다.

`git-gui`는 또한 파일의 특정 라인만 선택적으로 Staging할 수 있는 기능도 제공합니다. 예를 들어, `index.html` 파일에서 헤더 변경 사항만 Staging하고 푸터 변경 사항은 다음 커밋으로 미루고 싶다면, 변경된 코드 블록을 마우스로 선택한 후 "Stage This Hunk" 메뉴를 선택하면 됩니다. 이는 `git add -p` 명령어의 GUI 버전이라고 할 수 있습니다.

## 7. git-secrets: 보안 검사

개발 도구의 편의성도 중요하지만, 보안만큼 중요한 것은 없습니다. 특히 API 키나 비밀번호가 실수로 Git 저장소에 커밋되는 것은 치명적인 보안 사고로 이어질 수 있습니다.

`git-secrets`는 커밋에 비밀 키나 API 토큰이 포함되지 않도록 방지하는 도구입니다. AWS, GCP 등의 패턴이 기본 내장되어 있습니다.

`git-secrets`는 커밋이 생성되기 직전에 자동으로 검사를 수행합니다. 만약 커밋에 AWS Access Key, Secret Key, API Token, 비밀번호 등의 민감한 정보가 포함되어 있다면, 커밋을 차단하고 경고 메시지를 출력합니다. 이 도구는 특히 GitHub에 공개 저장소로 코드를 푸시할 때 필수적입니다. 한 번 커밋된 비밀 정보는 이력에서 완전히 제거하는 것이 매우 어렵기 때문에(모든 커밋 이력에서 해당 정보를 찾아 삭제해야 함), 사전에 차단하는 것이 최선의 방어책입니다.

```bash
# 설치
$ brew install git-secrets

# 저장소에 적용
$ git secrets --install

# AWS 키 패턴 등록
$ git secrets --register-aws
```

```text
$ git commit -m "config 추가"
   ┌─────────────────────────────────────────┐
   │  [ERROR]                                 │
   │  login.html:5:  AKIAIOSFODNN7EXAMPLE      │
   │                                          │
   │  ⚠ AWS Access Key ID found in commit!   │
   │                                          │
   │  파일에서 하드코딩된 키를 제거하거나     │
   │  git secrets --add 로 허용 패턴을 등록    │
   │  하세요.                                  │
   └─────────────────────────────────────────┘
   커밋이 차단되었습니다.
```

`git-secrets`는 기본적으로 AWS, GCP, Azure 등 주요 클라우드 서비스의 비밀 키 패턴을 내장하고 있습니다. 또한 사용자가 직접 커스텀 패턴을 추가할 수도 있습니다. 예를 들어, 회사 내부 API 키의 형식이 "COMPANY\_APIKEY\_"로 시작한다면, `git secrets --add 'COMPANY_APIKEY_[A-Z0-9]+'` 명령어로 패턴을 등록할 수 있습니다. 이렇게 등록된 패턴은 모든 커밋에서 자동으로 검사됩니다.

지금까지 우리가 살펴본 일곱 가지 도구는 각기 다른 용도와 장점을 가지고 있습니다. 아래의 한눈에 정리 표를 통해 각 도구의 특징과 설치 방법을 최종 정리하겠습니다.

## 한눈에 정리

| 도구 | 용도 | 설치 방법 |
|---|---|---|
| **tig** | 터미널 Git 브라우저 (히스토리 탐색) | `brew install tig` |
| **lazygit** | 터미널 Git GUI (직관적 조작) | `brew install lazygit` |
| **gh** | GitHub CLI | `brew install gh` |
| **diff-so-fancy** | diff 출력 가독성 향상 | `brew install diff-so-fancy` |
| **gitk** | GUI 히스토리 뷰어 (내장) | Git 설치 시 포함 |
| **git-gui** | GUI Staging/커밋 (내장) | Git 설치 시 포함 |
| **git-secrets** | 비밀 키 노출 방지 | `brew install git-secrets` |

## 연습 문제

이 장에서 소개한 도구들은 각각 특정 상황에서 Git 사용 효율성을 극대화해 줍니다. 아래 문제를 풀면서 각 도구의 특징과 적합한 사용 상황을 정리해 보시기 바랍니다.

1. `tig`를 설치하고 프로젝트의 커밋 히스토리를 탐색해보세요. `tig`와 `git log --oneline --graph --all`의 차이점은 무엇인가요?
2. `lazygit`를 설치하고 Staging, 커밋, 브랜치 전환을 마우스 없이 수행해보세요.
3. GitHub를 사용한다면 `gh`를 설치하고 `gh pr list` 명령어로 현재 열려있는 PR 목록을 확인해보세요.
4. `gitk`를 실행하고 그래프 뷰에서 브랜치 구조를 시각적으로 확인해보세요.
